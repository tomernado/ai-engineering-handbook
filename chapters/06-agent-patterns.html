פרק 6

Agent Patterns מתקדמים

פרק 5 הניח את התשתית: function calling, tool calling, ולולאת agent בסיסית. אבל "לולאה איטרטיבית" היא תיאור מבני בלבד — היא לא אומרת דבר על *איך* ה-agent מנמק בכל סבב, *מתי* הוא מתכנן מראש לעומת צעד-צעד, *אם* הוא בודק את עצמו, ו*איך* כמה agents מתואמים ביניהם כשיש יותר מאחד. הפרק הזה סוקר דפוסי עיצוב קונקרטיים שממלאים את הלולאה הכללית בתוכן מעשי — כל אחד פותר בעיה ספציפית שצצה כשמריצים agents על משימות אמיתיות: נימוק גלוי, תכנון מראש מול תכנון אגבי, ביקורת עצמית, תזמור בין-סוכני, נקודות עצירה לאדם, טיפול בכשלים, וזיכרון מעבר לריצה בודדת.

ReAct

Thought → Action → Observation, צעד אחד בכל סבב, בלי תוכנית מוקדמת.

Plan-and-Execute

תוכנית מלאה מראש, ואז ביצוע תתי-המשימות ברצף.

Reflexion

ביקורת עצמית על הפלט לפני מסירתו, ותיקון בתוך אותה ריצה.

שלושה דפוסים שממלאים את לולאת ה-Agent הכללית מפרק 5 בתוכן מעשי — כל אחד פותר בעיה שונה.

דפוס ReAct (Reasoning + Acting)

מה זה?

דפוס שבו המודל מנמק במפורש בטקסט (Thought) לפני כל פעולה (Action), צופה בתוצאה שחוזרת (Observation), ומשתמש בה כקלט לסבב הנימוק הבא — Thought → Action → Observation, שוב ושוב, עד שיש מספיק מידע לתשובה סופית.

למה צריך את זה?

Agent Loop הבסיסי מתאר את מבנה הבקרה (בקשה → כלי → תוצאה → החלטה) אבל לא כופה שום מבנה על ההחלטה עצמה. בלי נימוק גלוי, בחירת הפעולה הבאה היא "קופסה שחורה" — קשה לדבג למה agent בחר כלי מסוים, וקשה למודל עצמו לתקן מסלול שגוי כי אין רצף מחשבה מפורש שהוא יכול "להסתכל עליו" בסבב הבא. כתיבת Thought מפורש לפני Action משפרת עקביות ההחלטה וגם הופכת את התהליך לשקוף לניפוי שגיאות.

איך זה עובד?

ה-prompt מנחה את המודל להפיק בכל סבב שלושה חלקים בזה אחר זה:

  • Thought — נימוק מילולי על המצב הנוכחי ומה נדרש הלאה.
  • Action — קריאת הכלי הנגזרת מהנימוק.
  • Observation — תוצאת הכלי, שמוזנת בחזרה כהקשר לסבב הבא.

המחזור חוזר עד שה-Thought קובע שיש מספיק מידע, ואז מופקת תשובה סופית במקום Action נוספת.

דוגמה

בשאלה "כמה עובדים יש בחברה שרכשה את החברה שבה עבד X ב-2019", ה-Thought הראשון עשוי להיות "צריך קודם למצוא איפה עבד X ב-2019", ה-Action מפעילה כלי חיפוש, ה-Observation מחזירה את שם החברה, וה-Thought הבא בונה על התוצאה הזו כדי לנסח את שאילתת החיפוש הבאה — מסלול נימוק-פעולה שקוף שאפשר לעקוב אחריו צעד אחר צעד.

חזרה ל-Thought
אם נדרש מידע נוסף — סבב חדש
  • לשמור את ה-Thought בהיסטוריית השיחה, גם אם אינו מוצג למשתמש הסופי — הוא חיוני לדיבוג ולהמשכיות הנימוק בין סבבים.
  • לדרוש במפורש בפורמט הפלט הפרדה בין Thought ל-Action, כדי שקוד צד-לקוח יוכל לפרסר ולתעד כל שלב בנפרד.
טעויות נפוצות
  • להשמיט את שלב ה-Thought כדי "לחסוך טוקנים" — פעולה בלי נימוק גלוי נוטה יותר לבחירת כלי שגויה ואינה ניתנת לניפוי.
  • להתייחס ל-Observation כאל טקסט חופשי לא-מובנה שהמודל צריך "לנחש" לפרש, במקום להזין אותו בפורמט עקבי וברור.
AI Engineer Insight

ReAct הוא בעיקרו משמעת פורמט על גבי agent loop קיים, לא מנגנון חדש — אבל המשמעת הזו היא ההבדל בין agent שאפשר לדבג לבין agent שהתנהגותו אטומה. ברוב מסגרות ה-agent המודרניות (Claude Code, Cursor Agent Mode וכדומה) עקבות הנימוק גלויות בדיוק מהסיבה הזו.

מה חשוב לזכור

  • ReAct = Thought → Action → Observation, בלולאה, עם נימוק גלוי לפני כל פעולה.
  • נימוק גלוי משפר עקביות ודיבוג, במחיר טוקנים נוספים.

Plan-and-Execute

מה זה?

דפוס שבו שלב תכנון נפרד — פירוק המשימה המלאה לרשימת תתי-משימות מראש — קודם לשלב ביצוע, בניגוד ל-ReAct שמתכנן צעד בודד אחד בכל סבב בלי תוכנית כוללת מוקדמת.

למה צריך את זה?

תכנון צעד-אחר-צעד (כמו ב-ReAct) יעיל וגמיש, אבל במשימות ארוכות עם תלויות ברורות בין שלבים הוא מבזבז סבבי נימוק על גילוי מחדש של מבנה שכבר ידוע מראש. תוכנית מלאה מראש נותנת יעילות וזרימה ברורה — אפשר להריץ תתי-משימות בלי לנמק מחדש על המבנה הכולל בכל סבב. המחיר: פחות גמישות לשינויים באמצע — אם תת-משימה חושפת מידע שמפר הנחת יסוד של התוכנית, נדרש מנגנון תכנון-מחדש מפורש, לא רק "המשך הלאה".

איך זה עובד?

שלב תכנון ראשוני מפרק את הבקשה לרשימת תתי-משימות מסודרת (ולעיתים עם תלויות ביניהן). שלב ביצוע מריץ כל תת-משימה בזו אחר זו — כל אחת עשויה להפעיל agent loop משלה (ואף להיעזר ב-ReAct פנימית) — ותוצאות מצטברות. אם תת-משימה נכשלת או חושפת מידע סותר, נדרש צעד תכנון-מחדש מפורש שמעדכן את שאר התוכנית, ולא רק המשך עיוור לפי הרשימה המקורית.

דוגמה

כלים agentic לכתיבת קוד הפועלים במצב תכנון (כגון Claude Code או Cursor Agent Mode במצב "plan") מציגים תוכנית מפורשת — רשימת קבצים לשנות וסדר הפעולות — לפני שהם מתחילים לערוך, ורק לאחר אישור או תחילת ביצוע עוברים לשלב ריצה שמבצע כל צעד ברשימה. זה שונה מהותית מ-agent שמחליט צעד-צעד בלי לחשוף תוכנית כוללת מראש.

  • להשתמש ב-plan-and-execute כשהמשימה ארוכה ומבנה תתי-המשימות ידוע מראש ברמה גבוהה — כתיבת פיצ'ר רב-קבצים, מיגרציה, פייפליין מרובה-שלבים.
  • לבנות מנגנון תכנון-מחדש מפורש למקרה שתת-משימה חושפת מידע שסותר את התוכנית המקורית, ולא להניח שהתוכנית תמיד תקפה.
טעויות נפוצות
  • להשתמש בתוכנית קשיחה במשימה שבה כל צעד תלוי מהותית בתוצאת קודמו ואי-אפשר לחזות מראש — שם ReAct צעד-צעד מתאים יותר.
  • להמשיך ביצוע תוכנית אחרי שתת-משימה נכשלה, בלי לעצור ולהעריך אם שאר התוכנית עדיין רלוונטית.
AI Engineer Insight

הבחירה בין plan-and-execute ל-ReAct דומה לבחירה בין תכנון פרויקט מראש לבין פיתוח אג'ילי אינקרמנטלי — אין תשובה נכונה גורפת, יש טרייד-אוף בין יעילות-כשהמבנה ידוע לבין גמישות-כשהוא לא. agents רבים משלבים את שניהם: תוכנית ברמה גבוהה, וביצוע כל תת-משימה בלולאת ReAct פנימית.

מה חשוב לזכור

  • plan-and-execute = תכנון מלא מראש, ואז ביצוע; ReAct = תכנון צעד-צעד תוך כדי ריצה.
  • יעיל למשימות עם מבנה ידוע; פחות גמיש לשינויים באמצע.

Reflexion

מה זה?

דפוס שבו ה-agent מבקר את הפלט או הביצוע של עצמו (self-critique) לפני החזרת תוצאה סופית או לפני צעד הבא, ומזין את הביקורת בחזרה כהקשר — למידה מכישלון בתוך אותה ריצה, לא fine-tuning של המודל עצמו.

למה צריך את זה?

agent שמבצע פעולה ומקבל אותה כמוגמרת ללא בדיקה חוזרת, מעביר טעויות הלאה בלי הזדמנות לתקן. הוספת שלב ביקורת עצמית — "האם התוצאה הזו באמת עונה על הדרישה? מה חסר?" — לפני שהתוצאה נחשבת סופית, תופסת כשלים שהמודל עצמו יכול לזהות כשהוא בוחן את הפלט מחדש בפרספקטיבה נפרדת, גם בלי משוב חיצוני.

איך זה עובד?

לאחר הפקת פלט או ביצוע פעולה, סבב נוסף מפעיל את המודל (או instance נפרד שלו) בתפקיד מבקר: לבחון את התוצאה מול הדרישה המקורית ולזהות פערים או שגיאות. אם נמצאה בעיה, הביקורת מוזנת בחזרה כהקשר לסבב תיקון נוסף — לא איפוס והתחלה מחדש, אלא שיפור מבוסס על מה שכבר נעשה. המחזור חוזר עד שהביקורת עצמה קובעת שהתוצאה מספקת, או עד למגבלת סבבים.

דוגמה

agent שכתב פונקציה ומריץ עליה בדיקות שנכשלות אינו רק מקבל את הכישלון כפלט סופי — הוא קורא את פלט הבדיקה ככשל, מנמק מה בקוד גרם לו, ומתקן בהתאם, לפני שהוא מציג את הקוד כמוגמר. זו צורה מעשית של reflexion: הביקורת (פלט הבדיקה שנכשלה) מוזנת בחזרה לתוך אותה ריצה כדי לשפר את התוצאה.

  • להגדיר קריטריון ביקורת ברור וממוקד (בדיקות עוברות, עמידה בדרישה מפורשת) ולא רק "האם זה נראה טוב" — ביקורת מעורפלת מייצרת תיקונים חסרי כיוון.
  • להגביל את מספר סבבי הביקורת-תיקון, כמו כל לולאה — reflexion בלי עצירה עלולה להיכנס למחזור תיקונים אינסופי.
טעויות נפוצות
  • להתבלבל בין reflexion (למידה בתוך ריצה בודדת, נעלמת בסופה) לבין fine-tuning (עדכון קבוע של משקלי המודל) — הם פותרים בעיות שונות לחלוטין.
  • לדלג על שלב הביקורת במשימות קריטיות כדי לחסוך זמן ריצה, כשדווקא שם המחיר של שגיאה שלא נתפסה הוא הגבוה ביותר.
AI Engineer Insight

reflexion יעיל במיוחד כשיש אות בדיקה אובייקטיבי וזול — תוצאת בדיקה, שגיאת קומפילציה, ולידציה של schema — כי אז הביקורת אינה תלויה בשיפוט הסובייקטיבי של המודל על עצמו, אלא בעובדה חיצונית שהוא רק מנמק לגביה. reflexion ללא אות חיצוני נשען כולו על יכולת המודל לזהות את הטעויות של עצמו, שהיא מוגבלת יותר.

מה חשוב לזכור

  • reflexion = ביקורת עצמית בתוך הריצה, לא עדכון קבוע של המודל.
  • יעיל ביותר כשיש אות בדיקה אובייקטיבי (בדיקות, שגיאות קומפילציה) ולא רק שיפוט סובייקטיבי.

דפוסי תזמור Multi-Agent

מה זה?

שני דפוסים מנוגדים לתזמור מערכת multi-agent:

  • Supervisor — agent מרכזי מנתב משימות ל-agents מתמחים ומרכז את תוצאותיהם.
  • Swarm — agents שווי-מעמד המתקשרים ומעבירים שליטה זה לזה ישירות, ללא מתאם מרכזי.
למה צריך את זה?

ברגע שיש יותר מ-agent אחד, מישהו — או משהו — צריך להחליט מי פועל מתי ומי רואה את מה.

  • Supervisor — נותן שליטה ברורה ונקודת אחריות יחידה על הזרימה, אבל הופך לצוואר בקבוק כאשר כל תקשורת חייבת לעבור דרכו.
  • Swarm — מבזר את ההחלטה ומאפשר תקשורת ישירה בין agents שצריכים לשתף פעולה בצמידות, אך מוותר על נקודת בקרה מרכזית שמפשטת דיבוג ומעקב.
איך זה עובד?
  • Supervisor — agent מרכזי מקבל את הבקשה, מחליט לאיזה agent מתמחה לנתב אותה (או חלקים ממנה), מקבל את התוצאה חזרה, ומחליט אם לנתב לעוד agent, לרכז תשובה, או להחזיר למשתמש. כל agent מתמחה "מדבר" רק עם ה-Supervisor, לא ישירות עם agents אחרים.
  • Swarm — כל agent מחזיק יכולת להעביר שליטה ("handoff") ישירות ל-agent אחר כאשר הוא מזהה שהמשימה חורגת מתחום אחריותו, בלי לעבור דרך רכזת מרכזית — השליטה עוברת מיד ליד לפי הצורך שמתגלה תוך כדי ריצה.

דוגמה

מערכת תמיכת לקוחות עשויה לממש Swarm: agent קליטה שמזהה פנייה טכנית מעביר שליטה ישירות ל-agent טכני, וזה, אם מתגלה שהבעיה קשורה לחיוב, מעביר שליטה ל-agent חיוב — בלי לחזור דרך רכזת. לעומת זאת, מערכת פיתוח תוכנה שבה agent מתכנן מפרק משימה ומנתב תת-משימות ל-agent כתיבה ול-agent בדיקה, ומרכז את שתי התוצאות לפני מסירה למשתמש, היא דוגמה ל-Supervisor.

  • לבחור Supervisor כברירת מחדל כשנדרשת נקודת בקרה, לוג מרכזי, ואחריות ברורה על סדר הפעולות — קל יותר לדבג ולבקר.
  • לשקול Swarm כאשר agents צריכים תקשורת צמודה ותכופה זה עם זה, וניתוב הכל דרך מתאם מרכזי יוצר latency וצוואר בקבוק מיותרים.
טעויות נפוצות
  • לבחור Swarm כברירת מחדל "כי זה גמיש יותר" בלי להביא בחשבון שהיעדר נקודת בקרה מרכזית מקשה משמעותית על מעקב ודיבוג כשמשהו משתבש.
  • לבנות Supervisor שמעביר כל פרט מידע בין agents במקום להעביר רק את מה שרלוונטי לתת-המשימה — הופך את ה-Supervisor לצוואר בקבוק גם מבחינת עומס הקשר.
AI Engineer Insight

הבחירה בין Supervisor ל-Swarm היא בעצם אותה שאלה ארגונית שקיימת בצוותי אנוש: היררכיה עם מנהל פרויקט מרכזי, או צוות שטוח שמעביר בעיות ישירות בין חברים לפי מומחיות. אין תשובה "נכונה" אוניברסלית — יש טרייד-אוף בין נראות ובקרה (Supervisor) לבין מהירות ועקיפת צוואר בקבוק (Swarm), ומערכות רבות בפועל משלבות את שניהם ברמות שונות של המערכת.

מה חשוב לזכור

  • Supervisor = ניתוב וריכוז מרכזי; Swarm = handoff ישיר בין agents שווי-מעמד.
  • Supervisor קל יותר לדבג; Swarm מתאים לתקשורת צמודה בלי צוואר בקבוק.

Human-in-the-Loop

מה זה?

נקודות עצירה יזומות בתוך agent loop שבהן ה-agent ממתין באופן מפורש לאישור אדם לפני ביצוע פעולה בלתי-הפיכה או רגישה, במקום להמשיך אוטומטית.

למה צריך את זה?

לא כל טעות ניתנת לתיקון בסבב הבא — מחיקת נתונים, ביצוע תשלום, או שליחת הודעה ללקוח חיצוני הן פעולות שברגע שקרו, קרו. עבור פעולות מהסוג הזה שקיפות וניתנת-לתיקון (כמו reflexion) אינה מספיקה; נדרשת חסימה יזומה של הביצוע עד לאישור מפורש.

איך זה עובד?

כלים או פעולות מסווגים מראש כ"דורשי אישור"; כשה-agent מגיע לקריאה כזו, הלולאה נעצרת, הפעולה המוצעת מוצגת לאדם, וההרצה ממשיכה רק לאחר אישור (או מבוטלת/מותאמת לפי דחייה).

דוגמה

כלי agentic לכתיבת קוד עוצר ומציג diff לאישור לפני מחיקת קובץ או הרצת פקודת terminal הרסנית, אך ממשיך אוטומטית בפעולות הפיכות כמו קריאת קובץ או חיפוש בבסיס הקוד.

  • לסווג כלים מראש לפי הפיכות הפעולה, לא לפי תחושת בטן ברגע הריצה — הקריטריון צריך להיות ברור ועקבי מראש.
טעויות נפוצות
  • לדרוש אישור אדם על כל פעולה כולל הפיכות ולא-רגישות, מה שהופך את ה-agent לאיטי ומייתר את היתרון של אוטומציה.
AI Engineer Insight

human-in-the-loop יעיל ככל שהוא ממוקד — נקודת עצירה על כל פעולה הופכת לרעש שאדם מאשר בלי לקרוא; נקודת עצירה רק על הבלתי-הפיך נשארת משמעותית.

מה חשוב לזכור

  • עצירה יזומה לאישור אדם, ממוקדת בפעולות בלתי-הפיכות או רגישות בלבד.

טיפול בשגיאות בלולאת Agent

מה זה?

מנגנוני התמודדות עם כשלים בתוך agent loop: ניסיונות חוזרים (retries) עם השהיה גוברת (backoff), timeout לכלים שנתקעים, והחלטה מודעת מתי להיכשל בצורה מבוקרת במקום לנסות שוב ללא סוף.

למה צריך את זה?

כלים חיצוניים נכשלים — timeouts, שגיאות רשת זמניות, API עמוס. בלי מנגנון טיפול, כשל נקודתי עוצר את כל הריצה או, בקצה השני, agent שממשיך לנסות ללא הגבלה תוקע משאבים ומייצר עלות בלתי-צפויה.

איך זה עובד?

קריאת כלי שנכשלת מסווגת כזמנית (retry עם backoff מוגבל) או קבועה (fail gracefully — דיווח כשל ברור ל-LLM כדי שינמק על חלופה, במקום לחזור על אותה קריאה). timeout מוגדר לכל כלי כדי למנוע תקיעה בלתי-מוגבלת של הלולאה כולה.

דוגמה

קריאה ל-API חיצוני שנכשלת עם שגיאת timeout מנוסה שוב עד שלוש פעמים עם השהיה גוברת; אם כל הניסיונות נכשלים, ה-agent מקבל הודעת כשל מפורשת ("הכלי X אינו זמין") ומנמק על מסלול חלופי, במקום שהלולאה כולה תיתקע או תמשיך לנסות ללא הגבלה.

  • להגדיר timeout סביר לכל כלי ומגבלת retries קבועה מראש, לא ללא-הגבלה.
טעויות נפוצות
  • לתת לכל שגיאה, זמנית או קבועה, לעצור את כל הריצה במקום להבחין בין השתיים ולטפל בהתאם.
AI Engineer Insight

כשל מבוקר עם הודעה ברורה חזרה ל-LLM שקול לטיפול בחריגות בתוכנה רגילה — המטרה אינה למנוע כשל אלא לוודא שהוא מדווח בפורמט שהמערכת (או המודל) יכולה לנמק לגביו ולהגיב אליו.

מה חשוב לזכור

  • retries עם backoff לשגיאות זמניות; timeout לכלים תקועים; fail gracefully לכשל קבוע.

זיכרון Agent

מה זה?

שלושה סוגי זיכרון הרלוונטיים ל-agent:

  • short-term — ה-Context של הריצה הנוכחית בלבד, נעלם בסופה.
  • long-term — מידע כללי המאוחסן חיצונית ונטען לפי צורך בין ריצות נפרדות.
  • episodic — זיכרון של ריצות או משימות ספציפיות קודמות, לשימוש כתקדים במשימה עתידית דומה.
למה צריך את זה?

short-term בלבד אומר שה-agent "שוכח" הכל ברגע שהריצה מסתיימת — כל משימה מתחילה מאפס, גם אם דומה למשימה שכבר נפתרה. long-term ו-episodic מאפשרים לשמר ידע מעבר לריצה בודדת: long-term לעובדות ותצורה יציבה, episodic לתקדימים קונקרטיים ("איך נפתרה בעיה דומה בעבר") שיכולים לקצר ריצות עתידיות.

איך זה עובד?

short-term הוא פשוט ה-Context של הריצה — אין אחסון חיצוני, נגיש כל עוד הריצה פעילה. long-term ו-episodic דורשים אחסון חיצוני ומנגנון אחזור: מידע נכתב לאחסון בסיום ריצה (או תוך כדי), ובתחילת ריצה חדשה רלוונטית, נשלף ומוזרק ל-Context. ההבדל המעשי בין long-term ל-episodic הוא סוג המידע — long-term שומר עובדות/העדפות כלליות, episodic שומר תיאור של משימה ופתרונה כתקדים קונקרטי לשליפה כשמזוהה דמיון למשימה חדשה.

דוגמה

agent פיתוח שזוכר long-term ש"הפרויקט משתמש ב-TypeScript מחמיר" (עובדה יציבה, נטענת בכל ריצה) שונה מ-episodic memory שזוכר "בפעם הקודמת שנדרש להוסיף endpoint דומה, הפתרון כלל גם עדכון תיעוד API" (תקדים ספציפי, נשלף רק כשמזוהה משימה דומה).

  • להפריד באחסון בין long-term (עובדות יציבות) ל-episodic (תקדימי משימות), כדי שאחזור לא יערבב סוגי מידע שונים מהותית.
טעויות נפוצות
  • להסתמך רק על short-term ולצפות שה-agent "ילמד" בין ריצות בלי מנגנון אחסון חיצוני כלשהו — בלי long-term/episodic כל ריצה מתחילה מאפס.
AI Engineer Insight

סיווג הזיכרון כאן משלים, לא חופף, את ה-Memory Manager שנסקר בפרק 3: שם עסקנו במנגנון הטכני של ניהול חלון ה-Context (מה נשאר, מה מסוכם, מה נזרק) בתוך שיחה אחת; כאן ההבחנה היא ברמת סוג המידע ומשך חייו — בתוך ריצה, בין ריצות, או כתקדים ספציפי לשימוש עתידי.

מה חשוב לזכור

  • short-term = Context של הריצה הנוכחית; long-term = עובדות יציבות בין ריצות; episodic = תקדימי משימות ספציפיות.
  • long-term ו-episodic דורשים אחסון חיצוני ומנגנון אחזור — לא קורים מאליהם.

Real World Example

Claude Code / Cursor Agent Mode — שילוב plan-and-execute עם human-in-the-loop

כלי agentic לכתיבת קוד ממחיש שילוב טבעי של כמה מהדפוסים שנסקרו בפרק. הבקשה ("הוסף בדיקות אינטגרציה לפיצ'ר X") עוברת תחילה שלב plan-and-execute: הכלי מציג תוכנית — אילו קבצים ייבדקו, אילו תרחישים יכוסו — לפני שהוא נוגע בקוד. כל תת-משימה בתוכנית מתבצעת בלולאת ReAct פנימית: נימוק גלוי על הצעד הבא, קריאת כלי, בחינת התוצאה, וחזרה. כשמגיעה פעולה בלתי-הפיכה — מחיקת קובץ בדיקה ישן, למשל, או הרצת פקודת terminal שמשנה תצורה — הריצה עוצרת בנקודת human-in-the-loop ומחכה לאישור מפורש, בעוד עריכות קוד הפיכות ובדיקה של קבצים ממשיכות אוטומטית. אם קריאת כלי נכשלת (למשל timeout בהרצת בדיקות), מנגנון טיפול בשגיאות מנסה שוב עם backoff מוגבל לפני שהוא מדווח כשל מבוקר בחזרה ל-agent. לאחר כתיבת הבדיקות, שלב reflexion מריץ אותן ובוחן את התוצאה — אם בדיקה נכשלת, הכשל מוזן בחזרה כהקשר לתיקון, לא כסימן לוותר. ואם הפרויקט כבר נתקל בעבר בדפוס דומה, זיכרון episodic עשוי לספק תקדים רלוונטי שמקצר את הריצה. יחד, אלו לא פיצ'רים נפרדים אלא שכבות של אותה לולאת agent בסיסית מפרק 5 — מובנית, שקופה, וניתנת לעצירה במקום הנכון.
