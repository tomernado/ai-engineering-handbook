Memory & RAG

ה-LLM עצמו הוא stateless לחלוטין — כל קריאת API עומדת בפני עצמה, ללא זיכרון מובנה מקריאות קודמות וללא גישה לידע שלא נכלל באימונו. הפרק הזה עוסק בשני המנגנונים שמפצים על כך בשכבת האפליקציה: Memory, שמדמה המשכיות שיחה על ידי בחירה יזומה של היסטוריה לשליחה בכל קריאה; ו-RAG (Retrieval-Augmented Generation), שמזין למודל ידע עדכני וספציפי-לדומיין שלא היה חלק מהאימון, על ידי אחזור מסמכים רלוונטיים ברגע הבקשה.

Conversation Memory

מה זה?

המנגנון שבו ה-Backend שומר את היסטוריית השיחה בין המשתמש למערכת, ובוחר בכל קריאת API מחדש אילו הודעות מתוך ההיסטוריה הזו לצרף לבקשה הנוכחית כרשימת messages עם roles של User ו-Assistant (ראו System / User / Assistant Roles בפרק 2).

למה צריך את זה?

מכיוון שהמודל עצמו stateless, "זיכרון שיחה" הוא אשליה מהונדסת — אין תהליך רץ ברקע שזוכר. אם ה-Backend לא ישלח מחדש בכל קריאה את ההודעות הרלוונטיות מהשיחה עד כה, המודל יתייחס לכל הודעה כפנייה ראשונה ומבודדת, ללא כל מודעות למה שנאמר קודם. תחושת ה"שיחה הרציפה" שחווה משתמש בצ'אט היא תוצר של ניהול היסטוריה בצד השרת, לא יכולת מובנית של המודל.

איך זה עובד?

בכל פנייה חדשה של המשתמש, ה-Backend שולף את היסטוריית השיחה השמורה (במסד נתונים או במטמון), מוסיף אליה את ההודעה החדשה, ומעביר את הרצף כולו ל-Prompt Builder שמחליט אילו חלקים בפועל נכנסים לקריאה — לא בהכרח כל ההיסטוריה מהתחלה. תשובת המודל נשמרת בחזרה כהודעת Assistant בהיסטוריה, כדי שתהיה זמינה לקריאה הבאה. ככל שהשיחה מתארכת, שליחת כל ההיסטוריה גולמית הופכת יקרה ובלתי-מעשית — כאן נכנסות החלטות ניהול הקשר שנדונות בנושאים הבאים בפרק.

דוגמה

בצ'אט כמו זה של Claude או ChatGPT, כאשר משתמש שואל "ותן לי גם גרסה קצרה יותר" בהודעה חמישית בשיחה, המודל "מבין" למה הכוונה רק כי ה-Backend צירף מחדש את ארבע ההודעות הקודמות לקריאת ה-API הנוכחית. אם ההיסטוריה לא הייתה נשלחת, השאלה הייתה חסרת הקשר לחלוטין מבחינת המודל.

  • לשמור את היסטוריית השיחה בצד השרת (מסד נתונים / מטמון), לעולם לא להסתמך על "זיכרון" מצד המודל.
  • להתייחס לבחירת ההודעות הנשלחות כהחלטת הנדסה מפורשת, לא כברירת מחדל של "שלח הכול".
טעויות נפוצות
  • להניח שהמודל "זוכר" שיחות קודמות ללא שליחת היסטוריה מפורשת בכל קריאה.
  • לשלוח את כל היסטוריית השיחה ללא הגבלה בשיחות ארוכות, מה שמייקר כל קריאה ועלול לחרוג מ-Context Window.
AI Engineer Insight

הבנה שגויה נפוצה אצל מהנדסים חדשים בתחום היא לחשוב על "Memory" כתכונה של המודל. בפועל זו תכונה של הארכיטקטורה סביבו — כל ספק LLM, כולל Anthropic ו-OpenAI, מספק API stateless; ה"זיכרון" נבנה כולו באפליקציה שמעליו.

מה חשוב לזכור

  • המודל עצמו stateless — אין זיכרון מובנה בין קריאות API.
  • ה-Backend שומר היסטוריה ובוחר בכל קריאה מה לשלוח ממנה מחדש.
  • שיחה "רציפה" היא תוצר הנדסי, לא יכולת של המודל.

תזרים RAG

מה זה?

Retrieval-Augmented Generation (RAG) הוא תבנית ארכיטקטונית שבה, לפני קריאה ל-LLM, המערכת מאחזרת מסמכים רלוונטיים ממאגר ידע חיצוני ומצרפת אותם להקשר — כך שהמודל עונה בהתבסס על מידע ספציפי ועדכני, לא רק על מה שנלמד באימון.

למה צריך את זה?

ידע האימון של מודל קפוא בזמן (ראו Inference לעומת Training בפרק 1) ואינו כולל מידע פרטי, עסקי, או שהתעדכן לאחר מועד האימון. RAG פותר את הפער הזה ללא צורך לאמן מחדש את המודל: המידע הרלוונטי מסופק כטקסט בתוך ה-Context בזמן הבקשה, מה שגם מצמצם הזיות (ראו Hallucinations בפרק 1) כי המודל "מצטט" מתוך מקור נתון במקום להסתמך על שחזור מהזיכרון הפנימי שלו.

איך זה עובד?

שאלת המשתמש הופכת לוקטור embedding, הוקטור הזה משמש לחיפוש דמיון במסד נתונים וקטורי כנגד embeddings של מסמכים שמוזנו מראש, המסמכים הרלוונטיים ביותר נבחרים, ומצורפים כטקסט להקשר שנשלח ל-LLM יחד עם השאלה המקורית. חמשת השלבים — Question, Embedding, Vector Search, Relevant Documents, LLM — מתרחשים בכל בקשה מחדש, בזמן אמת.

דוגמה

עוזר תמיכה שנשאל "מה מדיניות ההחזרות שלכם למוצר שנרכש לפני חצי שנה?" לא "יודע" את התשובה מאימונו — הוא מאחזר את מסמך המדיניות הרלוונטי מהמאגר הפנימי של החברה, ומצרף אותו כהקשר לפני מתן התשובה. ללא RAG, המודל היה נאלץ לנחש או להזות תשובה כללית שאינה תואמת את המדיניות בפועל.

AI Engineer Insight

איכות תשובת RAG תלויה בשלב האחזור הרבה יותר מבשלב ה-Generation. מודל חזק שמקבל מסמכים לא-רלוונטיים ייתן תשובה גרועה; מודל חלש שמקבל את המסמך הנכון בדיוק לרוב יצליח. כשתשובת RAG לא מספקת, נקודת הבדיקה הראשונה היא שלב ה-Vector Search, לא ה-prompt.

מה חשוב לזכור

  • הרצף: Question → Embedding → Vector Search → Relevant Documents → LLM.
  • RAG מזין ידע עדכני/פרטי בזמן ריצה, ללא אימון מחדש של המודל.
  • איכות האחזור קובעת את איכות התשובה יותר מאיכות המודל עצמו.

Memory Manager

מה זה?

הרכיב האחראי להחליט, בכל קריאת API, אילו הודעות עבר או עובדות שנלמדו במהלך השיחה נכנסות בפועל ל-Context — באמצעות בחירת ה-N ההודעות האחרונות, סיכום (summarization) של היסטוריה ארוכה, או שילוב של השניים.

למה צריך את זה?

שיחה ארוכה שנשלחת גולמית וללא סינון חורגת מ-Context Window או הופכת יקרה מדי לכל קריאה בודדת. Memory Manager הוא מה שמאפשר לשיחה "להימשך" לאורך זמן בעלות סבירה.

איך זה עובד?

אסטרטגיה נפוצה: לשמור את ה-N ההודעות האחרונות במלואן, ולסכם (באמצעות קריאת LLM נפרדת) את כל מה שקדם להן לפסקת תקציר קצרה שמצורפת כהקשר רקע.

דוגמה

לאחר עשרות הודעות בשיחה תמיכה ארוכה, Memory Manager עשוי לשלוח למודל תקציר בן שתי שורות של הבעיה שדווחה בתחילת השיחה, בצירוף חמש ההודעות האחרונות במלואן — במקום את כל תמלול השיחה.

  • לסכם היסטוריה ישנה במקום להשמיט אותה לחלוטין, כדי לשמר רציפות עובדתית.
טעויות נפוצות
  • לחתוך היסטוריה ישנה בלי סיכום, ולאבד עובדות שהמשתמש מסתמך על כך שהמערכת "זוכרת".
AI Engineer Insight

סיכום היסטוריה הוא בעצמו קריאת LLM נוספת, ולכן כפוף לאותם סיכוני הזיה כמו כל פלט מודל אחר — תקציר שגוי עלול "להנציח" טעות לאורך כל המשך השיחה.

מה חשוב לזכור

  • Memory Manager מחליט מה מתוך ההיסטוריה נכנס להקשר בכל קריאה.
  • שילוב נפוץ: הודעות אחרונות מלאות + סיכום של מה שקדם להן.

Token Budgeting

מה זה?

חלוקה מכוונת ומוגבלת מראש של תקציב הטוקנים העומד לרשות קריאת API בין רכיביו: System Prompt, Memory (היסטוריית שיחה), RAG (מסמכים מאוחזרים), ורזרבה לתשובה הצפויה.

למה צריך את זה?

Context Window סופי, ומחיר קריאה נגזר ישירות ממספר הטוקנים הנשלחים (ראו תמחור בפרק 1). ללא תקציב מוגדר, רכיב אחד (למשל היסטוריה ארוכה) עלול "לבלוע" את כל המקום ולדחוק החוצה מידע קריטי כמו מסמכי RAG, ולפגוע גם באיכות וגם בעלות.

איך זה עובד?

Prompt Builder מקצה תקרת טוקנים לכל רכיב מראש (למשל: System קבוע, עד X טוקנים ל-Memory, עד Y ל-RAG), וכל רכיב שחורג מתקרתו נחתך או מסוכם לפני ההרכבה הסופית של הפרומפט.

דוגמה

קריאה עם Context Window של 128K עשויה להקצות כ-500 טוקנים ל-System Prompt, 2,000 להיסטוריית שיחה, 4,000 למסמכי RAG שאוחזרו, והשאר כרזרבה לתשובה — כדי שהוספת מסמך ארוך לא תדחוק החוצה את ההיסטוריה, ולהפך.

  • להגדיר תקרת טוקנים מפורשת לכל רכיב הקשר, לא רק תקרה כוללת לכל הקריאה.
טעויות נפוצות
  • להתעלם מתקציב טוקנים עד שקריאה נכשלת מחריגה מה-Context Window בפרודקשן.
AI Engineer Insight

תקציב טוקנים הוא כלי עלות לא פחות מכלי איכות — בקריאות בהיקף גדול, שינוי קטן בהקצאה (למשל צמצום מספר מסמכי RAG מ-5 ל-3) יכול לתרגם לחיסכון משמעותי מבלי לפגוע בתשובה.

מה חשוב לזכור

  • תקציב הטוקנים מתחלק בין System / Memory / RAG / תשובה צפויה.
  • הקצאה לא מבוקרת בין הרכיבים פוגעת גם באיכות וגם בעלות.

Embeddings

מה זה?

Embedding הוא וקטור של מספרים (בדרך כלל מאות עד אלפי ממדים) שמייצג את המשמעות הסמנטית של טקסט (או תוכן אחר), כך שטקסטים בעלי משמעות דומה מקבלים וקטורים הממוקמים קרוב זה לזה במרחב הרב-ממדי.

למה צריך את זה?

מחשבים לא יכולים להשוות "משמעות" של טקסט ישירות. embeddings הופכים משמעות למספרים שניתן להשוות ביניהם מתמטית — זו התשתית שמאפשרת חיפוש "לפי משמעות" (semantic search) במקום התאמת מילות מפתח בלבד, ועליה מבוסס כל תזרים RAG.

איך זה עובד?

מודל embedding ייעודי (נפרד ממודל השיחה) מקבל טקסט ומחזיר וקטור קבוע-אורך. מודלים אלו מאומנים כך שטקסטים הדומים מבחינה סמנטית — גם אם משתמשים במילים שונות לגמרי — יופקו לוקטורים קרובים במרחב, בעוד טקסטים שאינם קשורים יופקו לוקטורים רחוקים.

דוגמה

באפליקציית מתכונים כמו After Taste, ניתן להמחיש (כתרחיש היפותטי להדגמה, לא כתיאור פיצ'ר קיים בפועל) חיפוש מתכונים לפי embeddings של רשימת הרכיבים: וקטור של "עגבניות, בזיליקום, מוצרלה" יהיה קרוב במרחב הווקטורי לוקטור של "עגבניות שרי, גבינה איטלקית, עשבי תיבול" — גם ללא חפיפת מילים מדויקת — ורחוק מווקטור של "שוקולד, קמח, ביצים".

  • לבחור מודל embedding מתאים לשפה ולדומיין (למשל תמיכה בעברית אם הטקסטים בעברית).
  • לגזור embeddings מיחידות טקסט קטנות ומגובשות סמנטית, לא ממסמכים ארוכים שלמים.
טעויות נפוצות
  • להשוות embeddings שהופקו על ידי שני מודלים שונים — הוקטורים אינם באותו מרחב וההשוואה חסרת משמעות.
AI Engineer Insight

embedding אינו "תמצית" של הטקסט וניתן לקריאה אנושית ישירה — הוא ייצוג מתמטי המיועד להשוואה בין וקטורים. הערך שלו נובע כולו מהיחסים המרחביים בינו לבין embeddings אחרים, לא מהערכים הבודדים בתוכו.

מה חשוב לזכור

  • Embedding = ייצוג וקטורי של משמעות סמנטית.
  • טקסטים דומים במשמעות → וקטורים קרובים במרחב, גם ללא חפיפת מילים.
  • יש להשתמש באותו מודל embedding לאורך כל התהליך.

יסודות Vector DB

מה זה?

מסד נתונים וקטורי הוא מסד נתונים המתמחה באחסון embeddings וביצוע חיפוש דמיון (similarity search) מהיר ביניהם בקנה מידה — איתור, מתוך מיליוני וקטורים, אלו הקרובים ביותר לוקטור שאילתה נתון.

למה צריך את זה?

חיפוש דמיון מדויק (השוואת וקטור מול כל וקטור במאגר) הופך איטי מדי כשמדובר במאגר גדול. מסד נתונים וקטורי מספק מבני אינדוקס ייעודיים המאפשרים חיפוש דמיון קרוב-לוודאי (approximate) במהירות גבוהה, גם כשהמאגר מכיל מיליוני פריטים — זו התשתית שמאפשרת לשלב Vector Search של RAG לרוץ בזמן סביר.

איך זה עובד?

בשלב האינדוקס, כל מסמך (או קטע ממנו) מומר ל-embedding ונשמר במסד יחד עם המטא-דאטה שלו. בשלב השאילתה, שאלת המשתמש מומרת גם היא ל-embedding, ומסד הנתונים מחזיר את הפריטים שהוקטורים שלהם הכי קרובים לוקטור השאילתה, לפי מדד דמיון (ראו Cosine Similarity).

דוגמה

בתרחיש ההמחשה של After Taste, מסד נתונים וקטורי היה מאחסן את ה-embedding של כל אחד מ-66 המתכונים לפי רשימת הרכיבים שלו, ומאפשר לאחזר במהירות את המתכונים הדומים ביותר לשאילתת רכיבים חדשה — ללא צורך להשוות ידנית מול כל מתכון במאגר.

  • לאחסן לצד כל embedding מטא-דאטה שמאפשרת סינון (למשל תאריך, קטגוריה, מקור) לפני או אחרי חיפוש הדמיון.
טעויות נפוצות
  • לבחור מסד נתונים וקטורי לפי פופולריות בלבד, בלי לבדוק התאמה להיקף הנתונים ולדרישות התפעול בפועל.
AI Engineer Insight

ליסוד הזה בלבד — האם להשתמש במסד ייעודי, בתוסף וקטורי למסד קיים, או בפתרון מנוהל — יש שיקולי עלות, latency והתפעול שונים מהותית. השוואת מוצרים ספציפית מפורטת בהשוואת Vector DB בפרק 4.

מה חשוב לזכור

  • Vector DB מתמחה באחסון embeddings וחיפוש דמיון מהיר בקנה מידה.
  • חיפוש מבוסס אינדוקס קרוב-לוודאי, לא השוואה מדויקת מול כל פריט.

Cosine Similarity

מה זה?

מדד דמיון בין שני וקטורים, המחושב מהזווית ביניהם (קוסינוס הזווית) ולא ממרחקם המוחלט — ערך קרוב ל-1 מציין וקטורים כמעט זהים בכיוונם, ערך קרוב ל-0 מציין חוסר קשר.

למה צריך את זה?

ל-embeddings, כיוון הוקטור נושא את המשמעות הסמנטית, בעוד גודלו (magnitude) לרוב מושפע מגורמים כמו אורך הטקסט המקורי ואינו רלוונטי לדמיון. Cosine Similarity מתעלמת מהגודל ומודדת רק כיוון — ולכן היא המדד המקובל ביותר להשוואת embeddings, בהבדל ממדדי מרחק כמו מרחק אוקלידי הרגישים גם לגודל הוקטור.

איך זה עובד?

מסד הנתונים הוקטורי מחשב את Cosine Similarity בין וקטור השאילתה לבין כל וקטור מועמד (או קירוב מהיר שלו דרך האינדקס), ומחזיר את הפריטים בעלי הציון הגבוה ביותר כתוצאות החיפוש הרלוונטיות ביותר.

דוגמה

וקטור שאילתה של "רכיבים איטלקיים קלאסיים" יקבל ציון Cosine Similarity גבוה מול וקטור מתכון הפסטה ברוטב עגבניות, וציון נמוך מול וקטור מתכון עוגת שוקולד — הציון הזה, לא מרחק גיאומטרי גולמי, הוא שקובע את דירוג התוצאות.

  • לוודא שמסד הנתונים הוקטורי מוגדר להשתמש ב-Cosine Similarity (או מדד תואם) כברירת מחדל למודל ה-embedding הנבחר.
טעויות נפוצות
  • להשתמש במרחק אוקלידי גולמי כברירת מחדל בלי לבדוק אם המודל תומך גם ב-Cosine Similarity לתוצאות טובות יותר.
AI Engineer Insight

עבור embeddings מנורמלים (uniform-length), Cosine Similarity ומכפלה פנימית (dot product) מניבים אותו דירוג תוצאות — הבחירה בין השניים היא לרוב שיקול ביצועים טכני, לא הבדל איכותי בתוצאה.

מה חשוב לזכור

  • Cosine Similarity מודדת זווית בין וקטורים, לא מרחק מוחלט.
  • זהו המדד המקובל ל-embeddings כי הכיוון נושא את המשמעות הסמנטית.

RAG מול Fine-Tuning

מה זה?

שתי גישות שונות מהותית להתאמת מודל למשימה ספציפית:

  • RAG — מזין ידע חיצוני להקשר בזמן הבקשה מבלי לשנות את המודל עצמו.
  • Fine-Tuning — מאמן מחדש (חלקית) את משקלי המודל כך שהתנהגותו, סגנונו, או דפוסי הפלט שלו משתנים באופן קבוע.
למה צריך את זה?

בחירה שגויה בין השניים היא כשל ארכיטקטוני יקר:

  • Fine-Tuning על ידע שמתעדכן תדיר מחייב אימון מחדש בכל שינוי.
  • RAG לשינוי סגנון או פורמט קבוע דורש חזרה על אותה הנחיה בכל פרומפט ואינו אמין באותה מידה כמו התנהגות "צרובה" במודל.
איך זה עובד?

כלל ההכרעה המעשי:

  • RAG — כאשר הצורך הוא ידע — עובדות, מסמכים, מידע שמשתנה או מתעדכן — מכיוון שעדכון מאגר המסמכים מיידי וזול לעומת אימון מחדש.
  • Fine-Tuning — כאשר הצורך הוא שינוי התנהגות קבוע — טון, פורמט פלט עקבי, דפוס חשיבה ספציפי לדומיין — מכיוון שההתנהגות הרצויה הופכת חלק אינטגרלי מהמודל ואינה תלויה בהזכרה מחדש בכל פרומפט.
דוגמה

עוזר משפטי שצריך לענות על סמך תקנות שמתעדכנות מדי רבעון מתאים ל-RAG — עדכון מאגר המסמכים אינו דורש נגיעה במודל. עוזר שצריך תמיד להחזיר תשובות בפורמט משפטי-פורמלי קבוע ובסגנון כתיבה ספציפי ועקבי לחלוטין לאורך אלפי בקשות מתאים יותר ל-Fine-Tuning, כי ההתנהגות הופכת ברירת מחדל של המודל עצמו.

  • להתחיל תמיד מ-RAG ומ-Prompt Engineering לפני שיקול Fine-Tuning — עלות ומורכבות ההפעלה נמוכות משמעותית.
  • לשקול Fine-Tuning רק כאשר יש דפוס התנהגות עקבי שחוזר על עצמו ולא ניתן להשגה אמינה דרך prompt בלבד.
טעויות נפוצות
  • לבצע Fine-Tuning כדי "להזין" למודל עובדות חדשות — ידע עדכני שייך ל-RAG, לא לאימון מחדש.
  • לנסות לאכוף סגנון עקבי לחלוטין רק דרך System Prompt ארוך יותר ויותר, כשהפתרון היעיל בפועל הוא Fine-Tuning.
AI Engineer Insight

ברוב פרויקטי ה-production, RAG פותר את רוב הצרכים העסקיים הריאליים בעלות נמוכה משמעותית מ-Fine-Tuning — ולכן הוא ברירת המחדל הראשונה שיש לבחון. Fine-Tuning נשמר למקרים ספציפיים שבהם התנהגות עקבית קריטית ואינה מושגת אמינה דרך prompt. להרחבה על טכניקות RAG מתקדמות ראו פרק 4.

מה חשוב לזכור

  • ידע משתנה/עדכני → RAG; התנהגות/סגנון/פורמט קבוע → Fine-Tuning.
  • RAG זול ומהיר לעדכון; Fine-Tuning יקר אך "צורב" התנהגות עקבית.
  • יש להתחיל מ-RAG ומ-Prompt Engineering לפני שיקול Fine-Tuning.

Real World Example

Claude / ChatGPT — Memory ואחזור מקורות

עוזרים שיחתיים כמו Claude ו-ChatGPT ממחישים כיצד שני מנגנוני הפרק פועלים זה לצד זה. שכבת Conversation Memory שומרת את היסטוריית הצ'אט ומחליטה, בכל תגובה, אילו הודעות קודמות לצרף מחדש לקריאת ה-API — כולל, במוצרים מסוימים, שכבת Memory Manager שמסכמת שיחות ישנות או שומרת עובדות נבחרות לטווח ארוך במקום להזרים תמלול גולמי אינסופי. כאשר משתמש מבקש מידע עדכני או מבוסס-מקור — למשל שאלה שדורשת חיפוש רשת — המערכת מפעילה תבנית דמוית-RAG: היא מאחזרת תוצאות רלוונטיות ממקור חיצוני, מצרפת אותן להקשר, ורק אז מנסחת תשובה עם ציון המקורות שממנה נשאבה — בדיוק העיקרון שתואר בנושא תזרים RAG, מיושם כאן על תוכן חי מהאינטרנט במקום מאגר מסמכים סטטי.
