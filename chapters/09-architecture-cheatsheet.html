פרק 9

Architecture & Cheat Sheet

פרק זה הוא נספח ההפניה של הספר: מפת הארכיטקטורה המלאה של מערכת LLM בייצור, רשימת בדיקה לפני כל בקשה, מדריך בחירת מודל לפי קריטריונים, טבלת השוואת משפחות מודלים, ומילון מונחים מרוכז לכל הספר. אין כאן הסברים מחודשים — רק עיון מהיר.

ארכיטקטורה מלאה

מה זה?

מסלול חיים מלא של בקשה במערכת LLM production, מקצה לקצה — מרגע שהמשתמש שולח קלט ועד שהתשובה חוזרת אליו. כל שלב בתרשים מייצג שכבה אחרת בפרקים 1–8 של הספר.

User
שולח בקשה או שאלה בממשק המוצר.
Frontend
קולט את הקלט ומעביר לבקאנד, מציג בהמשך את התשובה (לרוב עם streaming).
Backend
מתאם את כל שלבי העיבוד, אכיפת מדיניות, ו-rate limiting.
Memory
טוען הקשר שיחה קודם רלוונטי למשתמש.
Vision
מעבד קלט חזותי אם קיים (תמונה, מסמך סרוק).
RAG / Tools
מאחזר מסמכים רלוונטיים ו/או מפעיל כלים חיצוניים לפי הצורך.
Prompt Builder
מרכיב את הפרומפט הסופי מכל המקורות — system, זיכרון, RAG, קלט.
AI Gateway
מנתב לספק/מודל המתאים, אוכף rate limiting ו-fallback.
LLM
מריץ inference ומחזיר תשובה (או קריאת כלי נוספת).
Validation
בדיקת תקינות טכנית ו-guardrails של מדיניות ובטיחות על הפלט.
Logging
תיעוד מלא של הקריאה לצורך דיבוג, ביקורת, ו-monitoring.
Frontend
מציג את התשובה הסופית למשתמש.
AI Engineer Insight

גם בקשה "פשוטה" למראה עוברת בפועל את רוב השכבות האלה — זיכרון, ניתוב gateway, validation, logging — גם אם המשתמש חווה רק שאלה ותשובה. ההבדל בין פרוטוטייפ ל-production הוא בדיוק היכולת לבנות, לתחזק, ולנטר את כל השרשרת הזו כמערכת אחת, לא רק את קריאת המודל בודדת שבמרכזה.

Cheat Sheet — לפני כל בקשה

למה צריך את זה?

שישה מרכיבים שכדאי לוודא במפורש לפני שליחת כל קריאת LLM — רשימת בדיקה מהירה, לא הסבר מחודש.

  • System Prompt — מוגדר בבירור, כולל תפקיד, מגבלות, וטון, ומעודכן לגרסה הנוכחית של המוצר.
  • User Prompt — הקלט מנוסח או מעובד בצורה שמפרידה אותו בבירור מהוראות המערכת.
  • Memory — רק הקשר שיחה רלוונטי ועדכני נטען, לא היסטוריה גורפת שמנפחת את ה-context לחינם.
  • RAG — אחזור המסמכים תואם את השאלה בפועל, ותוצאות שאינן רלוונטיות מסוננות לפני הכנסה ל-prompt.
  • Max Tokens — מוגבל לפי צורך אמיתי של סוג הבקשה, לא ברירת מחדל גבוהה גורפת.
  • Temperature — מותאם למשימה: נמוך למשימות עקביות ועובדתיות, גבוה יותר למשימות יצירתיות.

Cheat Sheet — החלטות ארכיטקטורה נפוצות

למה צריך את זה?

שמונה שאלות שחוזרות בכל תכנון מערכת LLM — מיפוי מהיר בין מצב לבחירה מומלצה, עם קישור לפרק המרחיב. זו טבלת עיון, לא הסבר מחודש של המושגים.

מצב → החלטה
המצב הבחירה המומלצת הרחבה
המודל צריך גישה למידע עדכני, פרטי, או שלא היה חלק מהאימון RAG פרק 3
צריך לשנות סגנון, פורמט, או התנהגות קבועה של המודל באופן עקבי בכל קריאה Fine-Tuning פרק 3
המודל צריך לבצע פעולה בעולם האמיתי — קריאה ל-API, הרצת קוד, שליפת נתון חי Tool Calling פרק 5
המשימה דורשת כמה צעדים תלויי-תוצאה, שהרצף המדויק ביניהם לא ידוע מראש Agent (לולאת Agent) פרק 5
יש טווח משימות ברמות מורכבות שונות, ורוצים לאזן עלות מול איכות Model Routing פרק 8
כמה agents מתמחים צריכים לשתף פעולה על אותה משימה דפוס תזמור — Supervisor או Swarm פרק 6
צריך לוודא שפלט המודל לא חורג ממדיניות תוכן או בטיחות לפני שהוא יוצא החוצה Guardrails פרק 8
צריך למדוד אם שינוי בפרומפט או במודל שיפר או פגע באיכות בפועל Evals על Golden Dataset פרק 7

מה חשוב לזכור

  • RAG ו-Fine-Tuning פותרים בעיות שונות — ידע מול התנהגות — ולעיתים קרובות נדרשים שניהם יחד.
  • Tool Calling הוא היכולת הטכנית הבסיסית; Agent הוא השימוש בה בתוך לולאה איטרטיבית מרובת-צעדים.
  • Model Routing ו-Orchestration פותרים בעיות שונות: הראשון מאזן עלות/איכות לכל קריאה, השני מתאם בין agents.
  • אף החלטה כאן אינה חד-פעמית — מערכת production בוגרת בדרך כלל משלבת כמה מהן יחד.

בחירת מודל — מדריך החלטה

איך זה עובד?

בחירת מודל נכונה מבוססת על קריטריונים ארכיטקטוניים, לא על שם מודל ספציפי שמתיישן במהירות:

  • Context window נדרש — כמה טקסט (מסמכים, היסטוריית שיחה, תוצאות כלים) חייב להיכנס לפרומפט בבת אחת.
  • רמת עלות — עלות לטוקן מוכפלת בנפח הקריאות בקנה מידה של ייצור, לא רק בעלות קריאה בודדת.
  • איכות tool-use / function-calling — עד כמה המודל אמין בבחירת הכלי הנכון ובמילוי פרמטרים תקינים, קריטי במיוחד למערכות agentic.
  • דרישות latency — האם המשימה דורשת תגובה כמעט-מיידית (ממשק שיחתי חי) או שיש סובלנות לזמן עיבוד ארוך יותר (עיבוד אצווה ברקע).
  • self-hosting / open-weight — נדרש כאשר יש דרישת residency של נתונים (המידע לא יכול לצאת מסביבה מבוקרת), או כשעלות בקנה מידה גדול הופכת אירוח עצמי לכדאי יותר מקריאת API.
דוגמה

משימת סיווג טקסט פשוטה בנפח גבוה (למשל תיוג פניות תמיכה נכנסות) מתאימה למודל קל ומהיר — עלות לטוקן נמוכה חשובה יותר מיכולת נימוק מתקדמת. משימת agent מורכבת רבת-שלבים (תכנון, קריאת כלים מרובה, קבלת החלטות) דורשת מודל חזק עם tool-use איכותי — כאן חיסכון בעלות על חשבון אמינות ההחלטות עלול לייצר כשלים שקטים ויקרים יותר בטווח הארוך.

מה חשוב לזכור

  • לבחור מודל לפי קריטריונים (context, עלות, tool-use, latency, self-hosting), לא לפי שם מותג.
  • עלות נמדדת בקנה מידה של נפח ייצור, לא בקריאה בודדת.
  • self-hosting רלוונטי כשיש דרישת residency של נתונים או כשההיקף הופך אותו לכדאי כלכלית.
  • ההשוואה בפועל בין מודלים זמינים מפורטת בטבלת Model Quick Reference.

Model Quick Reference

השוואת משפחות מודלים לפי מחלקת יכולת
משפחה חוזקות טיפוסיות Context Window טיפוסי מתי לבחור
מודלים מובילים (flagship) נימוק מורכב, tool-use אמין, ביצועים גבוהים במגוון רחב של משימות גדול משימות agentic רבות-שלבים, החלטות בעלות השפעה גבוהה, כתיבה/ניתוח מורכבים
מודלים קלים/מהירים (lightweight) latency נמוך, עלות לטוקן זולה משמעותית בינוני עד גדול סיווג, חילוץ שדה בודד, משימות בנפח גבוה שאינן דורשות נימוק עמוק
מודלים open-weight/self-hosted שליטה מלאה בסביבת ההרצה, ללא תלות בספק חיצוני משתנה לפי דגם דרישת residency נתונים, עלות בקנה מידה גדול, סביבות מבודדות/ללא חיבור לרשת חיצונית
מודלים מתמחי-reasoning ביצועים גבוהים במשימות נימוק רב-שלבי מפורש, מתמטיקה, תכנון גדול משימות שדורשות שרשרת היסק ארוכה ומדויקת, לרוב במחיר latency גבוה יותר

מילון מונחים

Agent
מערכת מבוססת LLM שמתכננת ומבצעת רצף פעולות באופן איטרטיבי, כולל קריאת כלים, לצורך השגת מטרה.
AI (Artificial Intelligence)
תחום העוסק בבניית מערכות שמבצעות משימות שבדרך כלל דורשות אינטליגנציה אנושית.
AI Gateway
שכבת ביניים מרכזית בין האפליקציה למודלים שונים, המספקת ניתוב, ניהול מפתחות, rate limiting, ו-fallback במקום אחד.
Chunking
חלוקת מסמכים ארוכים ליחידות (chunks) קטנות יותר לפני יצירת embeddings, לצורך אחזור יעיל ומדויק ב-RAG.
Context (Context Window)
כמות הטוקנים המקסימלית שמודל יכול לעבד בבת אחת בקריאה בודדת, כולל קלט ופלט יחד.
Embedding
ייצוג וקטורי מספרי של טקסט (או תוכן אחר) שבו קרבה במרחב הווקטורי משקפת קרבה סמנטית במשמעות.
Golden Dataset
אוסף מקרי בדיקה עם תשובות ייחוס מאומתות, המשמש כבסיס עקבי להערכת ביצועי מערכת LLM לאורך זמן.
Guardrails
מסננים ברמת מדיניות ובטיחות שרצים לפני ואחרי קריאת המודל, לחסימת תוכן פוגעני או חריגה ממדיניות.
Hybrid Search
שילוב חיפוש סמנטי (embeddings) עם חיפוש מבוסס מילות מפתח (keyword/BM25) לצורך שיפור דיוק האחזור.
Inference
תהליך הרצת המודל המאומן על קלט נתון לצורך הפקת פלט, בניגוד לשלב האימון עצמו.
LLM (Large Language Model)
מודל שפה גדול שאומן על נפחי טקסט עצומים לחיזוי הטוקן הבא, ומשמש כבסיס ליישומי AI מבוססי שפה.
LLM-as-Judge
שימוש במודל שפה נפרד להערכת איכות פלט של מודל אחר, כתחליף או השלמה להערכה אנושית.
MCP (Model Context Protocol)
פרוטוקול סטנדרטי לחיבור מודלים וסוכני AI למקורות מידע וכלים חיצוניים בממשק אחיד.
Model Routing
ניתוב דינמי בין מודלים שונים בהתאם למורכבות המשימה, עלות, ו-latency נדרשים.
Prompt
הקלט הטקסטואלי שנשלח למודל, לרוב מורכב מהוראות מערכת, הקשר, ובקשת המשתמש.
RAG (Retrieval-Augmented Generation)
טכניקה שבה מאחזרים מידע רלוונטי ממקור חיצוני ומזינים אותו לפרומפט לפני קריאת המודל, כדי לבסס תשובות על מידע עדכני או פרטי.
ReAct
תבנית פעולה עבור agents המשלבת נימוק מפורש (reasoning) ופעולה (acting) לסירוגין, איטרציה אחר איטרציה.
Reflexion
תבנית שבה agent מעריך את התוצאה של פעולותיו הקודמות ומשתמש בביקורת עצמית זו לשיפור הניסיון הבא.
Reranking
שלב נוסף באחזור RAG שמדרג מחדש תוצאות אחזור ראשוניות לפי רלוונטיות מדויקת יותר, לרוב באמצעות מודל ייעודי.
Semantic Cache
מנגנון caching שמחזיר תשובה שמורה עבור קלט הדומה סמנטית לקלט קודם, גם כשהניסוח המילולי שונה.
Streaming
מסירת תשובת המודל למשתמש בהדרגה, טוקן אחר טוקן, במקום המתנה לתשובה מלאה לפני הצגתה.
Token
יחידת הטקסט הבסיסית שמודל שפה מעבד — לרוב חלק ממילה, מילה שלמה, או סימן פיסוק.
Tool Calling
יכולת של מודל לבקש הפעלה של פונקציה או כלי חיצוני מוגדר מראש, ולקבל את תוצאתו כקלט להמשך העיבוד.
Vector DB
מסד נתונים המותאם לאחסון embeddings וחיפוש יעיל של הוקטורים הקרובים ביותר לשאילתה נתונה.
