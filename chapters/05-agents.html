Agents & Modern AI

עד כה עסקנו במודל שמקבל הקשר (פרק 2) ומרחיב אותו במידע חיצוני (פרקים 3-4), אך תמיד באותה תבנית: קלט אחד, תשובת טקסט אחת. הפרק הזה עוסק במעבר לתבנית שונה מהותית — מודל שיכול לפעול על העולם, לא רק לתאר אותו. function calling מספק את הבסיס הטכני שמאפשר למודל להחזיר קריאה מובנית במקום טקסט חופשי; tool calling מרחיב זאת לבחירה בין כלים מרובים; agent loop הופך את זה לתהליך איטרטיבי שבו המודל מתכנן, פועל, ומגיב לתוצאות בחזרה. לצד זאת נסקור MCP כפרוטוקול לחיבור כלים, vision כערוץ קלט נוסף מעבר לטקסט, ומודלי reasoning שמבצעים תהליך חשיבה פנימי לפני מתן תשובה.

Function Calling

מה זה?

יכולת של LLM להחזיר, במקום תשובת טקסט חופשי, פלט מובנה המייצג קריאה לפונקציה — שם הפונקציה ופרמטרים תואמי-schema — כדי שקוד צד-לקוח יריץ אותה בפועל. המודל עצמו אינו מריץ קוד; הוא רק מפיק את הבקשה המובנית, וקוד חיצוני מבצע אותה ומחזיר את התוצאה חזרה למודל.

למה צריך את זה?

טקסט חופשי אינו ניתן לביצוע אוטומטי באופן אמין — פענוח בקשה כמו "תבדוק את מזג האוויר בתל אביב" מתוך פרוזה חופשית דורש parsing שביר ורגיש לניסוח. function calling פותר את זה בשורש: המודל מאומן ו/או מונחה להפיק JSON תואם-schema קבוע מראש, כך שקוד צד-לקוח יכול לסמוך על המבנה ולהריץ אותו ישירות בלי היסק לשוני נוסף. זהו הבסיס הטכני שעליו נבנים tool calling ו-agents כולם — בלעדיו, שום פעולה מהימנה מצד המודל אינה אפשרית.

איך זה עובד?

קריאת ה-API כוללת הגדרת פונקציה — שם, תיאור, ו-schema של הפרמטרים (בדרך כלל JSON Schema). המודל מקבל את ההגדרה יחד עם הקלט, ובמקום להחזיר טקסט רגיל הוא עשוי להחזיר אובייקט מובנה שמזהה את שם הפונקציה ואת ערכי הפרמטרים שחילץ מהבקשה. קוד צד-לקוח מקבל את האובייקט הזה, מריץ את הפונקציה המתאימה בפועל (קריאת API חיצונית, שאילתה למסד נתונים, חישוב), ומחזיר את התוצאה למודל כהודעה נוספת בשיחה כדי שיוכל לנסח תשובה סופית בהתבסס עליה.

דוגמה

גם Claude וגם מודלי GPT תומכים ב-function calling כחלק מובנה מה-API: מפתח מגדיר פונקציה כמו get_weather(city: string), ובבקשה "מה מזג האוויר בתל אביב?" המודל מחזיר קריאה מובנית get_weather(city="Tel Aviv") במקום לנחש תשובה מתוך הידע שנצבר באימונו — כך התשובה מבוססת על נתון עדכני אמיתי שהתקבל מקריאה בפועל.

  • להגדיר schema מדויק וממוקד לכל פונקציה — שמות פרמטרים ברורים ותיאורים מפורטים משפרים משמעותית את דיוק החילוץ של המודל.
  • להחזיר תמיד את תוצאת הפונקציה כהודעה בשיחה, גם כאשר הביצוע נכשל — כישלון שקוף עדיף על היעדר תגובה.
טעויות נפוצות
  • להניח שהמודל עצמו מריץ את הפונקציה — הוא רק מפיק את הבקשה המובנית; ביצוע בפועל הוא באחריות הקוד הצד-לקוח.
  • להגדיר schema עמום או פרמטרים ללא תיאור, מה שמוביל לחילוץ פרמטרים שגוי או חסר.
AI Engineer Insight

function calling הוא נקודת המפנה שממנה LLM הופך מ"מחולל טקסט" ל"רכיב תוכנה שאפשר לשלב בפייפליין". איכות ה-schema קובעת ישירות את אמינות המערכת — schema מעורפל מייצר קריאות שגויות בשקט, לא שגיאות גלויות, ולכן קשה לאתר בזמן ריצה.

מה חשוב לזכור

  • function calling = פלט מובנה (שם + פרמטרים) במקום טקסט חופשי; הביצוע עצמו קורה בקוד צד-לקוח.
  • schema מדויק וברור משפר משמעותית את דיוק החילוץ.
  • זהו הבסיס הטכני ל-tool calling ול-agents.

Tool Calling

מה זה?

הרחבה מעשית של function calling: המודל מקבל רשימת "כלים" זמינים, כל אחד עם schema משלו, ובוחר בעצמו איזה כלי (אם בכלל) להפעיל ומתי — כחלק מתהליך רב-שלבי, לא כקריאה בודדת.

למה צריך את זה?

משימות אמיתיות דורשות שילוב של יכולות שונות — חיפוש, חישוב, קריאה למסד נתונים — והמודל צריך להחליט לבד אילו מהן רלוונטיות לבקשה נתונה, בסדר הנכון.

איך זה עובד?

כל הכלים מוגדרים מראש בקריאת ה-API; המודל בוחר כלי (או אף אחד, אם תשובת טקסט ישירה מספיקה), מקבל את התוצאה, ולעיתים בוחר כלי נוסף בהתבסס עליה — לולאה שמתוארת בהרחבה בנושא Agent Loop.

דוגמה

כלים agentic לכתיבת קוד כמו Cursor ו-GitHub Copilot מקבלים ערכת כלים — קריאת קובץ, חיפוש בבסיס קוד, הרצת פקודת terminal — ובוחרים דינמית איזה כלי מפעילים בכל שלב בהתאם למשימה.

  • להגביל את מספר הכלים הזמינים בכל קריאה למינימום הרלוונטי — עומס כלים גדול מדי פוגע בדיוק הבחירה.
טעויות נפוצות
  • לחשוף עשרות כלים בבת אחת בלי סינון לפי הקשר המשימה.
AI Engineer Insight

tool calling הוא function calling פלוס שיקול דעת — ההבדל בין "המודל יודע להפיק JSON" לבין "המודל יודע מתי כדאי להשתמש בו".

מה חשוב לזכור

  • tool calling = בחירה בין כלים מרובים כחלק מתהליך רב-שלבי.
  • עומס כלים גדול מדי פוגע בדיוק הבחירה של המודל.

Agent Loop

מה זה?

מבנה הבקרה שהופך tool calling מקריאה חד-פעמית לתהליך איטרטיבי: בקשת המשתמש עוברת תכנון, הפעלת כלים, קריאה חוזרת ל-LLM עם התוצאות, וחזרה על הצעד הזה עד שיש מספיק מידע לתשובה סופית.

למה צריך את זה?

משימות מורכבות אינן נפתרות בקריאת כלי בודדת — לעיתים תוצאת כלי אחד חושפת שנדרש כלי נוסף, או שהתוצאה חלקית ודורשת ניסוח מחדש. לולאה מאפשרת למודל להגיב לתוצאות ביניים במקום להתחייב מראש לרצף פעולות קבוע.

איך זה עובד?

הבקשה נכנסת, שלב תכנון (מפורש או מרומז בתוך המודל עצמו) קובע איזו פעולה נדרשת, כלי מופעל, התוצאה חוזרת ל-LLM כהקשר נוסף, וה-LLM מחליט אם יש מספיק מידע לתשובה סופית או שנדרש סבב נוסף בלולאה. התהליך נעצר כשה-LLM מפיק תשובת טקסט סופית במקום קריאת כלי נוספת, או כשמגיעים למגבלת איטרציות שהוגדרה מראש.

דוגמה

בבקשה "תקן את הבאג בטיפול בשגיאות אימות", כלי agentic לכתיבת קוד יבצע כמה סבבים בלולאה: חיפוש בבסיס הקוד למציאת הקוד הרלוונטי, קריאת הקובץ, ניתוח הבעיה, עריכה, ולבסוף הרצת בדיקות לאימות התיקון — כל שלב מזין את הבא, לא רצף קבוע מראש.

  • להגדיר מגבלת איטרציות מקסימלית ללולאה, כדי למנוע ריצה אינסופית או עלות בלתי-מבוקרת.
  • להזין למודל את תוצאות הכלי בפורמט ברור וממוקד, כדי שההחלטה על הצעד הבא תתבסס על מידע נקי.
טעויות נפוצות
  • להתייחס ל-agent כאל קריאת כלי בודדת ולצפות שהוא יפתור משימה רב-שלבית בצעד אחד.
  • להריץ לולאה ללא מגבלת עצירה, מה שעלול לגרור עלות וזמן ריצה בלתי-צפויים במקרה של תקיעות לוגית.
AI Engineer Insight

ההבדל המהותי בין "מודל עם כלים" ל"agent" הוא הלולאה עצמה — לא מספר הכלים הזמינים. מערכת עם כלי בודד שמופעל פעם אחת אינה agent; מערכת שיודעת להגיב לתוצאותיה שלה ולהחליט על צעד הבא היא כן, גם אם יש לה רק כלי אחד זמין.

מה חשוב לזכור

  • הלולאה: User → Planner → Tools → LLM → Response, עם חזרה כשצריך.
  • זו לולאה איטרטיבית, לא קריאה חד-פעמית — ההחלטה על הצעד הבא נעשית בכל סבב.
  • מגבלת איטרציות חובה כדי למנוע ריצה בלתי-מבוקרת.

מבוא ל-Multi-Agent

מה זה?

מערכות שבהן כמה agents מתמחים משתפים פעולה על אותה משימה, במקום agent כללי בודד — לדוגמה agent אחד שמתכנן, אחד שמבצע, ואחד שבודק את התוצאה.

למה צריך את זה?

agent בודד עם prompt כללי מדי נוטה להתפזר במשימות מורכבות ורב-תחומיות. פיצול לתפקידים מתמחים — כל אחד עם prompt, כלים, ולעיתים אף מודל מותאמים לתפקידו — משפר מיקוד ואיכות, במחיר מורכבות תזמור נוספת.

איך זה עובד?

כל אחד רץ כתהליך עצמאי יחסית עם תחום אחריות מוגדר:

  • Agent מתכנן — מפרק את המשימה לתת-משימות.
  • Agent מבצע — מריץ אותן (כולל agent loop משלו).
  • Agent בודק — מאמת את התוצאה לפני מסירתה למשתמש או חזרה לתיקון.
דוגמה

במשימת פיתוח תכונה מורכבת, agent מתכנן עשוי לפרק אותה לצעדים, agent מבצע כותב את הקוד לכל צעד, ו-agent בודק מריץ בדיקות ומחזיר משוב לפני שהתוצאה נחשבת מוכנה.

  • להפריד תפקידים כאשר יש עדות שסוכן כללי בודד מתפזר או מפספס שלבי אימות במשימה מורכבת.
טעויות נפוצות
  • לפצל למספר agents משימה פשוטה שסוכן בודד היה פותר באותה איכות, ולהוסיף מורכבות תזמור שלא לצורך.
AI Engineer Insight

multi-agent אינו ברירת מחדל — הוא פתרון לבעיה ספציפית של פיזור תשומת לב במשימות רב-תחומיות, ומגיע במחיר תזמור ותקשורת בין-סוכנית. להרחבה על דפוסי תזמור multi-agent ראו פרק 6.

מה חשוב לזכור

  • multi-agent = agents מתמחים משתפים פעולה (למשל: מתכנן, מבצע, בודק).
  • משפר מיקוד ואיכות במשימות מורכבות, במחיר מורכבות תזמור.

MCP — Model Context Protocol

מה זה?

פרוטוקול אחיד לחיבור כלים ומקורות מידע חיצוניים למודלים, במקום אינטגרציה מותאמת-אישית לכל שילוב כלי-מודל בנפרד.

למה צריך את זה?

בלי פרוטוקול אחיד, חיבור M כלים ל-N מודלים שונים דורש עד M×N אינטגרציות נפרדות — כל כלי חדש דורש עבודת אינטגרציה מול כל מודל, וכל מודל חדש דורש עבודה מול כל כלי קיים. MCP הופך את זה לבעיה לינארית: כלי שמדבר MCP זמין לכל מודל שתומך ב-MCP, וכל מודל שתומך ב-MCP יכול לדבר עם כל כלי כזה, ללא קוד אינטגרציה ייעודי לכל זוג.

איך זה עובד?

שרת MCP חושף ממשק אחיד — רשימת כלים, מקורות מידע, ו-schema מוגדר — וכל לקוח (סביבת agent, כלי עריכה, אפליקציה) שתומך בפרוטוקול יכול להתחבר אליו ולהשתמש בכלים שהוא חושף באותה צורה, בלי קשר לאיזה מודל מריץ את הבקשה בפועל.

דוגמה

שרת MCP שחושף גישה למסד נתונים ארגוני נכתב פעם אחת, ואז זמין לכל סביבת agent תומכת-MCP — בין אם מריצה אותה Claude או מודל אחר — בלי לכתוב שכבת אינטגרציה נפרדת לכל שילוב.

  • לחשוף כלים ומקורות מידע פנימיים דרך שרת MCP אחד במקום אינטגרציה נקודתית לכל סביבת agent בנפרד.
טעויות נפוצות
  • לבנות אינטגרציה ייעודית לכל שילוב כלי-מודל כשקיים כבר שרת MCP רלוונטי, ולשכפל עבודה שלא לצורך.
AI Engineer Insight

MCP פותר בעיה ארגונית, לא רק טכנית: ברגע שכלי פנימי חשוף כשרת MCP, כל צוות בארגון שמפעיל agents יכול לצרוך אותו מבלי לתאם קוד אינטגרציה מול הצוות שבנה את הכלי.

מה חשוב לזכור

  • MCP הופך M×N אינטגרציות לבעיה לינארית — פרוטוקול אחד במקום התאמה לכל זוג כלי-מודל.
  • שרת MCP נכתב פעם אחת וזמין לכל לקוח תומך-פרוטוקול.

Vision — קלט חזותי

מה זה?

יכולת של מודל multimodal לעבד תמונה כקלט לצד טקסט — לקדד את התוכן החזותי לייצוג שהמודל יכול לנמק לגביו יחד עם שאר ההקשר של השיחה.

למה צריך את זה?

הרבה משימות מתחילות בקלט שאינו טקסטואלי מטבעו — צילום מסך של שגיאה, תרשים, טופס סרוק. תיאור מילולי של תמונה על ידי המשתמש מאבד מידע ודורש מאמץ; קלט חזותי ישיר מאפשר למודל לנמק ישירות על התוכן כפי שהוא.

איך זה עובד?

התמונה עוברת קידוד ויזואלי — רכיב ייעודי שממיר את הפיקסלים לייצוג שהמודל השפתי יכול לעבד. הייצוג הזה משולב לתוך ה-Context יחד עם הטקסט, כך שהמודל מנמק על שניהם באותו תהליך הסקה אחד, ולא כשני שלבים נפרדים.

מודלי Reasoning

מה זה?

מודלים בעלי "חשיבה מורחבת" (extended thinking / reasoning tokens) המבצעים תהליך חשיבה פנימי — פירוק הבעיה, בדיקת חלופות, תיקון עצמי — לפני הפקת התשובה הסופית המוצגת למשתמש.

למה צריך את זה?

בעיות מורכבות רב-שלביות — הוכחה מתמטית, ניתוח לוגי מרובה-תנאים, תכנון עם תלויות — נהנות מתהליך חשיבה מפורש שמאפשר למודל לבדוק ולתקן את עצמו לפני שהוא "מתחייב" לתשובה. עבור משימות פשוטות ועובדתיות, אותו תהליך חשיבה מוסיף latency ועלות ללא שיפור תואם באיכות התשובה — היתרון תלוי-משימה, לא גורף.

איך זה עובד?

לפני הפקת התשובה הגלויה, המודל מייצר רצף "מחשבה" פנימי שאינו בהכרח מוצג למשתמש במלואו, ורק לאחר מכן מנסח תשובה סופית מבוססת על תהליך הביניים הזה. הבחירה להפעיל תהליך כזה, ובאיזו עצימות, ניתנת לרוב לשליטה — כדי לאזן בין איכות ל-latency בהתאם לצורך במשימה הספציפית.

דוגמה

יכולות extended thinking (למשל אצל Claude) ומודלי סדרת-o (למשל אצל OpenAI) מייצגות גישה כללית דומה: הקצאת "תקציב חשיבה" גדול יותר למשימות שדורשות נימוק רב-שלבי, לעומת תגובה מיידית למשימות פשוטות שבהן חשיבה מורחבת רק מוסיפה זמן המתנה מיותר.

  • להפעיל reasoning מורחב עבור משימות רב-שלביות שבהן טעות בשלב אחד פוגמת בתוצאה הסופית — קוד מורכב, ניתוח לוגי, תכנון.
  • לבחון אמפירית האם reasoning מורחב אכן משפר תוצאה במשימה נתונה לפני הפעלתו כברירת מחדל, ולא להניח שהוא תמיד עדיף.
טעויות נפוצות
  • להפעיל reasoning מורחב כברירת מחדל על כל בקשה, כולל שאלות עובדתיות פשוטות שאינן זקוקות לו, ולשלם latency ועלות ללא תועלת.
  • להתייחס לתהליך החשיבה הפנימי כאל תשובה סופית מוצגת — הוא שלב ביניים, לא הפלט שנועד למשתמש.
AI Engineer Insight

ההחלטה מתי להפעיל מודל reasoning דומה במבנה לטרייד-אוף retrieve-then-rerank מפרק 4 — משאב חישובי יקר יותר לא כברירת מחדל גורפת, אלא כשלב מודע שמופעל היכן שהמורכבות בפועל מצדיקה אותו.

מה חשוב לזכור

  • מודלי reasoning מבצעים תהליך חשיבה פנימי לפני התשובה הסופית.
  • משתלם בבעיות מורכבות רב-שלביות; מיותר ויקר-latency במשימות פשוטות.
  • עוצמת החשיבה ניתנת בדרך כלל לכיוונון לפי צורך המשימה.

Real World Example

Cursor Agent Mode / GitHub Copilot Workspace — לולאת agent מלאה

מצב agentic בכלי כתיבת קוד כמו Cursor או GitHub Copilot Workspace ממחיש כיצד כל נושאי הפרק פועלים יחד. הבקשה ("הוסף endpoint חדש שמאמת טוקן ומחזיר פרופיל משתמש") נכנסת ללולאת Agent Loop: שלב תכנון מפרק אותה לתת-משימות — איתור קובץ ה-routes הרלוונטי, בדיקת מנגנון האימות הקיים, כתיבת הקוד החדש, הרצת בדיקות. כל תת-משימה מופעלת דרך tool calling — קריאת קובץ, חיפוש בבסיס קוד, כתיבת קובץ, הרצת terminal — כאשר כל קריאה בנויה על מנגנון function calling הבסיסי. אם צילום מסך של שגיאת בדיקה מצורף, הכלי מעבד אותו דרך ערוץ vision ומשלב אותו בהקשר לצד הקוד. במשימות שדורשות נימוק רב-שלבי מורכב — למשל תכנון מחדש של ממשק בין שכבות — הכלי עשוי להפעיל reasoning מורחב לפני שהוא מציע שינוי. תוצאת כל שלב חוזרת ללולאה, עד שהבדיקות עוברות והתשובה הסופית — קוד עובד, לא רק הצעה — מוצגת למשתמש. זהו ההבדל בין "השלמת קוד" לבין agent במובן המלא: לא צעד בודד, אלא תהליך איטרטיבי שמגיב לתוצאותיו שלו.
