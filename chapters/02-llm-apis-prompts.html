LLM APIs ו-Prompt Engineering

פרק 1 הסביר מה קורה בתוך המודל. הפרק הזה עובר לשכבת האפליקציה: איך בקשה בודדת ממשתמש הופכת לקריאת API בנויה היטב, מהי הנוסחה שמניבה prompt איכותי, ואיך להבטיח שהפלט שחוזר מהמודל בטוח לשימוש במערכת production — לא רק "נשמע טוב".

ארכיטקטורת בקשת API

מה זה?

בקשה בודדת ממשתמש לפיצ'ר מבוסס LLM אינה קריאת API יחידה שקורית "לבד" — היא עוברת רצף שכבות מוגדר:

  • Frontend — אוסף את קלט המשתמש.
  • Backend — מטפל בלוגיקת האפליקציה ובאימות.
  • Prompt Builder — מרכיב את הפרומפט הסופי משילוב של הנחיות קבועות וקלט דינמי.
  • LLM API — מבצע את ה-Inference עצמו.
  • שכבת Validation — בודקת את הפלט לפני שהוא יוצא.
  • Frontend — מציג את התוצאה למשתמש.
למה צריך את זה?

הפרדת השכבות הזו היא מה שהופך פרויקט "קריאה ל-API של Claude" לארכיטקטורת production אמיתית. ללא Prompt Builder נפרד, כל שינוי בהנחיות מחייב לגעת בקוד עסקי; ללא שכבת Validation, כל תקלה במודל (JSON פגום, שדה חסר, הזיה) זולגת ישירות למשתמש או למסד הנתונים. חשיבה במונחי pipeline — לא "קריאת API בודדת" — היא מה שמאפשר תחזוקה, בדיקות, וניפוי תקלות שיטתי.

איך זה עובד?

Frontend שולח בקשת HTTP ל-Backend עם קלט גולמי של המשתמש (לא נשלח ישירות ל-LLM API מסיבות אבטחה ומפתחות API). ה-Backend מאמת הרשאות ותקינות קלט, ומעביר את המידע הרלוונטי ל-Prompt Builder — רכיב שאחראי להרכיב את הפרומפט הסופי משילוב של System Prompt קבוע, תבניות (templates), והקשר דינמי (ראו נושא Context Engineering). הפרומפט הבנוי נשלח ל-LLM API, שמחזיר טקסט או JSON גולמי. שכבת Validation בודקת מבנה, טיפוסים, ולעיתים גם תוכן, ורק פלט שעבר אימות מוחזר ל-Frontend להצגה.

דוגמה

באפליקציית מתכונים כמו After Taste, פיצ'ר "הוסף מתכון מטקסט חופשי" עובר את כל השרשרת: המשתמש מדביק טקסט חופשי ב-Frontend, ה-Backend שולח אותו ל-Prompt Builder שמצרף הנחיות קבועות לחילוץ שדות מתכון, ה-LLM מחזיר JSON, שכבת Validation מוודאת שכל שדה חובה (שם, רכיבים, הוראות) קיים ומהסוג הנכון לפני שמירה ב-Supabase.

  • להפריד את Prompt Builder לרכיב עצמאי הניתן לבדיקה ולשינוי ללא נגיעה בלוגיקה עסקית.
  • להתייחס לכל שכבה כנקודת כשל אפשרית עצמאית, ולתעד לוגים בכל מעבר בין שכבות.
טעויות נפוצות
  • לקרוא ל-LLM API ישירות מה-Frontend, מה שחושף מפתחות API וגלוי לשינוי prompt על ידי המשתמש.
  • לערבב את בניית הפרומפט בתוך קוד הבקרים (controllers) העסקיים, מה שמקשה על תחזוקה ובדיקות.
  • להחזיר פלט LLM ישירות ל-Frontend ללא שכבת Validation מפורשת ביניים.
AI Engineer Insight

ככל שהמערכת גדלה, שכבת Prompt Builder הופכת לנכס הנדסי בפני עצמו — גרסאות פרומפט, A/B testing בין ניסוחים, ולוגים של הפרומפט המלא שנשלח בכל קריאה הופכים חיוניים לניפוי תקלות. מערכת ללא הפרדה כזו לא תוכל לענות על השאלה "מה בדיוק נשלח למודל שגרם לתשובה הזו".

מה חשוב לזכור

  • הרצף: Frontend → Backend → Prompt Builder → LLM API → Validation → Frontend.
  • LLM API נקרא רק מה-Backend, לעולם לא מה-Frontend.
  • Prompt Builder ו-Validation הם רכיבים עצמאיים, לא קוד מוטמע.

System / User / Assistant Roles

מה זה?

כל קריאת API ל-LLM שיחתי בנויה כרשימת הודעות עם תפקיד (role) מוגדר לכל אחת:

  • System — מגדיר את ההתנהגות הכללית והקבועה של המודל.
  • User — מכיל את קלט המשתמש בפועל.
  • Assistant — מכיל תשובות קודמות של המודל עצמו (לצורך שמירת היסטוריית שיחה).
למה צריך את זה?

ההפרדה בין הרולים אינה רק סמנטית — היא שכבת האמון היחידה שקיימת ל-API. הנחיה שנשלחת כ-System נחשבת מהימנה ובעלת עדיפות; טקסט שנשלח כ-User (ובפרט טקסט שמקורו במשתמש קצה או במסמך חיצוני) חייב להיחשב פוטנציאלית עוין. בלבול בין הרולים — למשל הזרמת תוכן חיצוני לא-מהימן לתוך ה-System Prompt — פותח פרצת prompt injection.

איך זה עובד?

ה-API מקבל מערך הודעות מסודר; המודל מתייחס לתוכן כל role באופן הבא:

  • System — בעל משקל גבוה יותר בקביעת ההתנהגות הכללית.
  • User — הבקשה הנוכחית שיש להגיב אליה.
  • Assistant — הקשר שיחה קודם שיש לשמור עליו עקבי.
דוגמה

בבוט תמיכה, ה-System Prompt קובע "ענה רק בנושאי המוצר, לעולם אל תחשוף מידע פנימי"; הודעות ה-User הן שאלות הלקוח בפועל. הזנת תוכן ממייל לקוח ישירות כ-System Prompt במקום כ-User תהיה טעות ארכיטקטונית שמפרה את גבול האמון.

  • להזרים תוכן ממקור לא-מהימן (קלט משתמש, מסמכים חיצוניים) תמיד תחת role של User, לעולם לא System.
  • לשמור את ה-System Prompt קבוע ותמציתי, נפרד מהתוכן הדינמי.
טעויות נפוצות
  • להזרים טקסט ממקור חיצוני לתוך ה-System Prompt, מה שמעניק לו אמון-יתר ופותח סיכון אבטחה.
AI Engineer Insight

הבחנת הרולים היא קו ההגנה הראשון נגד prompt injection — לא פרט מימוש טכני שולי. מהנדס שמזרים תוכן לא-מהימן תחת role לא נכון פותח פרצת אבטחה, לא רק בעיית פורמט.

מה חשוב לזכור

  • System = הנחיות מהימנות וקבועות; User = קלט נוכחי; Assistant = היסטוריה.
  • תוכן לא-מהימן תמיד תחת User, לעולם לא System.

נוסחת Prompt טוב

מה זה?

נוסחה מעשית לבניית prompt איכותי המורכבת מארבעה רכיבים:

  • Role — מיהו המודל בהקשר הזה.
  • Task — מה בדיוק נדרש לבצע.
  • Constraints — מגבלות פורמט, אורך, טון, איסורים.
  • Context — מידע רקע נדרש לביצוע המשימה.
למה צריך את זה?

prompt שנכתב כבקשה חופשית וכללית ("תעזור לי עם המתכון הזה") משאיר למודל להחליט החלטות פורמט וטון שהאפליקציה צריכה לשלוט בהן. נוסחה מובנית הופכת כתיבת prompt מאמנות אינטואיטיבית לתהליך הנדסי חוזר, הניתן לבדיקה, לגרסאות (versioning), ולשיפור שיטתי.

איך זה עובד?

כל רכיב ממלא תפקיד שונה בצמצום מרחב הפלט האפשרי:

  • Role — ממסגר את "קול" ואת נקודת המבט של התשובה.
  • Task — קובע את הפעולה הספציפית ומונע תשובות עמומות מדי.
  • Constraints — חוסמות פלטים לא-רצויים מראש (אורך, שפה, פורמט, מה אסור לכלול).
  • Context — מספק את המידע העובדתי הנדרש שאין למודל בלעדיו.

ארבעת הרכיבים יחד, לא כל אחד בנפרד, הם שמייצרים תשובה צפויה ורלוונטית.

דוגמה

Role: "את/ה עורך/ת תוכן קולינרי מקצועי/ת." Task: "המר את הטקסט החופשי הבא למתכון מובנה." Constraints: "החזר JSON בלבד לפי הסכמה הנתונה, ללא טקסט נוסף; כמויות במידות מטריות." Context: "הטקסט המקורי שהמשתמש הדביק: [טקסט המתכון]." שילוב ארבעת הרכיבים מייצר קריאת API יחידה שמניבה JSON תקין ועקבי, בניגוד ל"תמיר את זה למתכון" הכללי.

  • לכתוב כל אחד מארבעת הרכיבים כסעיף מפורש ונפרד בפרומפט, לא כמשפט אחד מעורבב.
  • להגדיר Constraints באופן חיובי ("החזר JSON בלבד") לצד שלילי ("אל תוסיף הסברים") כשרלוונטי.
  • לגרסאות (version) פרומפטים כמו קוד — שינוי בניסוח משפיע ישירות על איכות הפלט בפרודקשן.
טעויות נפוצות
  • לוותר על הגדרת Role, מה שמשאיר את "קול" התשובה בלתי-צפוי בין קריאות.
  • להשמיט Constraints כשהפלט מיועד לפרסינג אוטומטי, ולקבל טקסט חופשי במקום פורמט קבוע.
AI Engineer Insight

ברוב הכשלים בפרודקשן שמיוחסים ל"המודל לא מבין", הבעיה בפועל היא רכיב חסר בנוסחה — לרוב Constraints. הוספת שורה מפורשת המגדירה פורמט פלט פותרת חלק ניכר מהתקלות הללו בלי צורך בשינוי מודל.

מה חשוב לזכור

  • נוסחה: Role + Task + Constraints + Context.
  • כל רכיב מצמצם את מרחב הפלט האפשרי בדרך שונה.
  • Prompt הוא נכס הדורש גרסאות, לא טקסט חד-פעמי.

Context Engineering

מה זה?

הדיסציפלינה של בחירה, סינון וארגון המידע שמוזרם בפועל לתוך ה-Context (ראו Context Window בפרק 1) בכל קריאת API — היסטוריית שיחה, תוצאות אחזור, פלטי כלים, ומסמכי רקע — בנפרד מניסוח ה-prompt עצמו.

למה צריך את זה?

כתיבת prompt קובעת מה מבקשים מהמודל; Context Engineering קובעת עם אילו נתונים הוא עובד. שני הכשלים הנפוצים ביותר בפרודקשן — תשובה לא רלוונטית ותשובה יקרה מדי — נובעים כמעט תמיד מניהול הקשר לקוי, לא מניסוח prompt גרוע. זו הסיבה שהתחום הפך לתפקיד הנדסי נפרד, נבדל מ"prompt engineering" הצר.

איך זה עובד?

Prompt Builder (ראו נושא #1) אחראי בפועל ליישום Context Engineering: הוא מחליט אילו חלקי היסטוריה לכלול, אילו קטעי מידע לאחזר ולצרף, ובאיזה סדר למקם אותם (ראו lost-in-the-middle בפרק 1) — כל זאת לפני שהטקסט הסופי נשלח למודל. ההחלטות הללו מתעדכנות דינמית מבקשה לבקשה, בניגוד ל-System Prompt שנשאר קבוע.

דוגמה

עוזר קוד כמו Cursor או GitHub Copilot לא שולח את כל הריפו לכל קריאה — הוא בוחר את הקבצים הפתוחים, את הבלוק שבו הסמן נמצא, ותוצאות חיפוש רלוונטיות בלבד. זו החלטת Context Engineering, נפרדת לחלוטין מניסוח ההנחיה למודל.

  • לטפל בבחירת ההקשר כשלב הנדסי נפרד מכתיבת נוסח ה-prompt.
  • למדוד בפועל אילו רכיבי הקשר משפרים תשובה ואילו רק מייקרים קריאה.
טעויות נפוצות
  • להתמקד רק בליטוש ניסוח ה-prompt כשהבעיה בפועל היא הקשר לא רלוונטי או עודף.
AI Engineer Insight

שיפור תשובה של LLM לרוב מגיע יותר משינוי מה נכנס להקשר מאשר משינוי ניסוח ההנחיה עצמה. כשתשובה לא מספקת, השאלה הראשונה צריכה להיות "מה בדיוק נשלח בפועל בקריאה הזו" — לא "איך לנסח אחרת".

מה חשוב לזכור

  • Context Engineering = מה נכנס להקשר; Prompt Engineering = איך מנוסחת הבקשה.
  • Prompt Builder הוא הרכיב שמיישם את שתיהן יחד.
  • רוב תקלות האיכות בפרודקשן מקורן בהקשר, לא בניסוח.

Zero-Shot, Few-Shot ו-Chain-of-Thought

מה זה?

שלוש טכניקות ניסוח משימה למודל:

  • Zero-Shot — מבקש ביצוע ישיר ללא דוגמאות.
  • Few-Shot — מצרף מספר דוגמאות קלט-פלט לפני הבקשה עצמה כדי להדגים את הפורמט או ההיגיון הרצוי.
  • Chain-of-Thought (CoT) — מבקש מהמודל לפרט את שלבי ההיגיון בטקסט לפני מתן התשובה הסופית.
למה צריך את זה?

בחירת הטכניקה הנכונה משפיעה ישירות על איכות, עלות, ולטנסי:

  • Few-Shot — מיותר ומייקר משימות טריוויאליות; היעדרו במשימות עם פורמט לא-סטנדרטי מוביל לפלט לא-עקבי.
  • CoT — משפר דיוק במשימות היגיון רב-שלבי אך מכפיל טוקנים ולעיתים אינו נחוץ כלל למשימות פשוטות.
איך זה עובד?
  • Zero-Shot — מתאים למשימות שהמודל "מכיר" היטב מתוך האימון (סיווג פשוט, סיכום כללי) — אין רווח בהוספת דוגמאות.
  • Few-Shot — יעיל כשצריך לאכוף פורמט פלט ספציפי או סגנון לא-סטנדרטי: 2–5 דוגמאות מייצגות בדרך כלל מספיקות, ועודף דוגמאות מבזבז טוקנים ללא תועלת נוספת.
  • Chain-of-Thought — יעיל במשימות הדורשות שרשור לוגי (חישוב, ניתוח רב-שלבי) — הבקשה המפורשת "חשוב שלב אחר שלב לפני התשובה הסופית" משפרת דיוק כי היא מרחיבה את "שטח החישוב" הטקסטואלי לפני שהמודל מחויב לתשובה.
דוגמה

סיווג סנטימנט של ביקורת קצרה — Zero-Shot מספיק. חילוץ שדות מתכון לפורמט JSON ספציפי לאפליקציה — Few-Shot עם 2–3 דוגמאות ממחיש בדיוק את המבנה הרצוי. חישוב עלות כוללת של הזמנה עם הנחות מדורגות — CoT מבטיח שהמודל "יעבוד" את החישוב שלב-שלב במקום לנחש תוצאה סופית ישירות.

  • להתחיל תמיד מ-Zero-Shot ולהוסיף Few-Shot רק כשהפלט לא עקבי בפועל.
  • לבחור דוגמאות Few-Shot מייצגות את מלוא טווח המקרים, כולל מקרי קצה.
  • להפעיל CoT במשימות היגיון רב-שלבי בלבד — לא כברירת מחדל גורפת.
טעויות נפוצות
  • להוסיף עשרות דוגמאות Few-Shot "ליתר ביטחון", מה שמייקר כל קריאה ללא שיפור מדיד באיכות.
  • לבקש CoT יחד עם דרישת פלט JSON קשיח ללא הפרדה — שרשור המחשבה "מזהם" את הפלט המובנה.
AI Engineer Insight

כאשר נדרשים גם CoT וגם פלט מובנה, יש להפריד לשני שלבים: קריאה ראשונה עם CoT חופשי לקבלת הנמקה, קריאה שנייה (או פרסינג) שמחלצת רק את המסקנה הסופית לפורמט הקשיח. ניסיון לשלב את שניהם בקריאה אחת פוגע לרוב באמינות ה-JSON.

מה חשוב לזכור

  • Zero-Shot כברירת מחדל; Few-Shot לאכיפת פורמט; CoT להיגיון רב-שלבי.
  • כל תוספת דוגמאות/הנמקה מגדילה עלות וטוקנים — לא חינמית.
  • CoT ופלט מובנה קשיח לא תמיד משתלבים היטב באותה קריאה.

Structured Output

מה זה?

הכפפת פלט המודל לסכמה מוגדרת מראש במקום טקסט חופשי — באמצעות JSON mode, function-calling schemas (המודל "בוחר" לקרוא לפונקציה עם ארגומנטים מובנים), או אימות מול מודל טיפוסים כגון Pydantic (Python) או Zod (TypeScript).

למה צריך את זה?

כאשר הפלט מיועד לצריכה על ידי מערכת (מסד נתונים, API, קוד) ולא לקריאה אנושית, טקסט חופשי הוא ייצוג לא-אמין: כל שינוי קל בניסוח שובר parsing מבוסס regex. Structured Output הופך את הפלט לחוזה (contract) בין המודל למערכת, וניתן לאמת אותו אוטומטית לפני שימוש.

איך זה עובד?

הבקשה ל-API כוללת סכמה (JSON Schema או מקבילה) המתארת שדות, טיפוסים, ושדות חובה. חלק מהספקים אוכפים את הסכמה ברמת ה-API עצמו (garantia מבנית); באחרים יש לבקש זאת בפרומפט ולאמת בצד הלקוח. לאחר קבלת הפלט, ספריות כמו Pydantic או Zod מפרסרות ומאמתות את ה-JSON מול מודל טיפוסים מוגדר בקוד — כשל אימות נתפס מיד ולא מגיע לשכבות הבאות.

דוגמה

עוזרי קוד כמו GitHub Copilot ו-Cursor מפעילים עריכות קוד באמצעות פלט מובנה (diff/patch מובנה או function-calling), לא בקשה לכתוב "תתקן את זה" בטקסט חופשי — כך העריכה ניתנת ליישום אוטומטי, לביקורת (diff view), ולביטול (undo) מדויק. סכמה דומה נדרשת בפיצ'ר הוספת מתכון ב-After Taste: הפלט חייב להתאים למבנה טבלת המתכונים ב-Supabase.

{
  "type": "object",
  "properties": {
    "title": { "type": "string", "minLength": 2 },
    "category": {
      "type": "string",
      "enum": ["מנה עיקרית", "קינוח", "סלט", "מרק", "מאפה"]
    },
    "servings": { "type": "integer", "minimum": 1 },
    "ingredients": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "name": { "type": "string" },
          "amount": { "type": "number" },
          "unit": { "type": "string" }
        },
        "required": ["name", "amount", "unit"]
      },
      "minItems": 1
    },
    "steps": {
      "type": "array",
      "items": { "type": "string" },
      "minItems": 1
    }
  },
  "required": ["title", "category", "servings", "ingredients", "steps"],
  "additionalProperties": false
}
  • להעדיף JSON mode או function-calling האכוף ברמת ה-API כשהספק תומך, על פני בקשה בטקסט חופשי "החזר JSON".
  • להגדיר additionalProperties: false ושדות חובה מפורשים בסכמה כדי לצמצם דריפט פלט.
  • לאמת כל פלט מול מודל טיפוסים בקוד (Pydantic/Zod) לפני כל שימוש נוסף.
טעויות נפוצות
  • להסתפק בבקשת "החזר JSON בלבד" בטקסט חופשי בלי אכיפת סכמה, ולקבל לעיתים טקסט מקדים או Markdown עוטף.
  • להניח שסכמה בפרומפט מבטיחה תאימות מוחלטת — יש לאמת בכל זאת בקוד.
AI Engineer Insight

Structured Output מצמצם דרסטית את שכיחות כשלי parsing, אך אינו מבטל את הצורך בשכבת Validation נפרדת (ראו נושא הבא) — סכמה תקנית מבטיחה מבנה, לא נכונות תוכן (למשל כמות שלילית או קטגוריה שאינה קיימת בפועל בעסק).

מה חשוב לזכור

  • JSON mode, function-calling ו-Pydantic/Zod הן שלוש דרכי אכיפה משלימות.
  • סכמה עם שדות חובה ו-additionalProperties: false מצמצמת דריפט.
  • מבנה תקין ≠ תוכן נכון — Validation עדיין נדרש.

Output Validation

מה זה?

שכבת בדיקה טכנית שרצה על כל פלט LLM לפני שהוא נכתב למסד נתונים או מוזן למערכת production — מוודאת JSON תקין (parse מצליח), קיום כל השדות הנדרשים, וטיפוסי נתונים נכונים (מחרוזת, מספר, enum תקין).

למה צריך את זה?

לעולם אין להכניס פלט LLM ישירות למסד נתונים או למערכת production ללא אימות. גם עם Structured Output, פלט המודל נשאר תוצר הסתברותי (ראו Hallucinations בפרק 1) שעלול לחרוג מהסכמה, להכיל ערך לא-הגיוני, או להיכשל בפורמט. Validation היא רשת הביטחון הדטרמיניסטית היחידה סביב רכיב הסתברותי מטבעו.

איך זה עובד?

שכבת Validation פועלת בשלושה שלבים:

  • (1) parse — ניסיון לפענח את המחרוזת כ-JSON תקין, עם טיפול בכשל parse כשגיאה מפורשת ולא כניסיון "לתקן" את הטקסט.
  • (2) schema check — השוואת המבנה שהתקבל מול מודל הטיפוסים (Pydantic/Zod), כולל שדות חובה וטווחי ערכים.
  • (3) בדיקות עסקיות — נוספות בהתאם לתחום, למשל שהקטגוריה קיימת ברשימת הקטגוריות המותרות בפועל, לא רק שהיא מחרוזת.

פלט שנכשל באחד השלבים נדחה או מופעל מחדש (retry) עם הודעת שגיאה חוזרת למודל, ולא מגיע למשתמש או למסד הנתונים כפי שהוא.

דוגמה

בפיצ'ר הוספת מתכון ב-After Taste, גם אם ה-LLM מחזיר JSON תקני מבחינה תחבירית, שכבת Validation חייבת לוודא בנוסף ש-category הוא אחת מחמש הקטגוריות הקיימות בפועל באפליקציה ולא ערך שהמודל "המציא", ושה-servings הוא מספר חיובי הגיוני — לפני כתיבה ל-Supabase.

  • להריץ Validation דטרמיניסטי (קוד, לא LLM נוסף) כשלב חובה לפני כל כתיבה למסד נתונים.
  • להגדיר אסטרטגיית retry מפורשת (עם הודעת שגיאה חוזרת למודל) לפלט שנכשל באימות, במקום כשל שקט.
  • להבחין בין בדיקת מבנה טכני (Validation, הנושא הזה) לבין בדיקת מדיניות ובטיחות תוכן (Guardrails — ראו פרק 8).
טעויות נפוצות
  • לכתוב פלט LLM גולמי למסד נתונים production בלי שכבת אימות ביניים.
  • לנסות "לתקן" JSON פגום באמצעות מניפולציית מחרוזות (regex) במקום לדחות ולבקש פלט חדש.
  • לבלבל בין Validation טכני (הנושא כאן) לבין Guardrails מדיניות/בטיחות — שני מנגנונים נפרדים בתפקידם.
AI Engineer Insight

Output Validation אינה "בונוס" הנדסי אלא הגבול הדטרמיניסטי היחיד סביב רכיב הסתברותי. כל מערכת שמזרימה פלט LLM למסד נתונים או ללקוח production ללא שכבה זו חשופה לא רק לתקלות, אלא לכשלי שקט — נתונים פגומים שנכנסים למערכת בלי שאף התראה תישלח.

מה חשוב לזכור

  • לעולם לא להכניס פלט LLM גולמי ישירות למסד נתונים או ל-production.
  • שלושה שלבים: parse → schema check → בדיקות עסקיות.
  • Validation = נכונות טכנית; Guardrails = מדיניות ובטיחות (פרק 8).

Prompt Chaining

מה זה?

שרשור מספר קריאות LLM עוקבות, כאשר פלט קריאה אחת משמש כקלט לקריאה הבאה, לעומת ניסיון לפתור את כל המשימה בקריאה מורכבת אחת עם prompt ארוך ורב-שלבי.

למה צריך את זה?

משימה יחידה מורכבת (למשל "חלץ מתכון, תרגם אותו, וסווג לקטגוריה") שנדחסת ל-prompt אחד מגדילה סיכון לכשל חלקי בלתי-ניתן לניפוי — קשה לדעת איזה שלב נכשל. שרשור מפצל את המשימה לקריאות ממוקדות שכל אחת ניתנת לבדיקה, ל-Validation, ולטיפול שגיאות בנפרד.

איך זה עובד?

כל קריאה בשרשרת מקבלת prompt ממוקד ופלט מאומת מהשלב הקודם, ומבצעת שלב אחד בלבד. העלות היא לטנסי ועלות API גבוהים יותר (מספר קריאות במקום אחת); התועלת היא דיוק גבוה יותר לכל שלב, יכולת ניפוי תקלות ממוקדת (מזהים בדיוק איזה שלב נכשל), ואפשרות להשתמש במודל שונה או בפרמטרי דגימה שונים לכל שלב לפי צרכיו.

דוגמה

חילוץ מתכון מטקסט חופשי יכול להתפצל לשרשרת: קריאה ראשונה מחלצת שדות גולמיים, קריאה שנייה מנרמלת יחידות מידה, קריאה שלישית מסווגת לקטגוריה — לעומת prompt יחיד שמנסה לבצע את כל השלושה בבת אחת ומקשה על איתור מקור טעות בפלט הסופי.

  • לפצל לשרשרת כשמשימה כוללת מספר שלבים לוגיים נפרדים הדורשים דיוק גבוה בכל אחד.
  • להריץ Validation בין חוליות השרשרת, לא רק בסוף.
טעויות נפוצות
  • לשרשר קריאות למשימה פשוטה שמודל אחד פותר באיכות מספקת בקריאה בודדת, ולשלם עלות ולטנסי מיותרים.
AI Engineer Insight

ההחלטה בין prompt יחיד לשרשרת היא trade-off בין עלות/לטנסי לבין דיוק/ניתנות-לניפוי — לא כלל אצבע קבוע. משימה שנכשלת בעקביות בקריאה בודדת היא המועמדת הטבעית לפיצול, לא כל משימה מורכבת מראש.

מה חשוב לזכור

  • שרשור = פלט קריאה אחת הוא קלט לקריאה הבאה.
  • יתרון: דיוק וניפוי תקלות ממוקד; חיסרון: עלות ולטנסי גבוהים יותר.
  • לפצל כשמשימה יחידה נכשלת בעקביות, לא כברירת מחדל.

Real World Example

GitHub Copilot / Cursor

עוזרי קוד כמו GitHub Copilot ו-Cursor ממחישים כיצד שלושה מושגי הפרק פועלים יחד במערכת production. כל בקשת עריכה מנוסחת פנימית לפי נוסחת ה-prompt (Role: "עוזר תכנות מומחה בשפת הפרויקט", Task: "בצע את השינוי המבוקש", Constraints: "החזר diff מובנה בלבד", Context: קטעי הקוד הרלוונטיים שנבחרו על ידי Context Engineering — לא הריפו כולו). התשובה חוזרת כ-Structured Output — patch/diff מובנה, לא טקסט חופשי — ועוברת Validation טכני (בדיקת syntax, קיום הקבצים המוזכרים) לפני שהעריכה מוצגת למשתמש להשוואה וקבלה. כשמדובר בשינוי רב-קבצים, המערכת עשויה להפעיל Prompt Chaining — שלב זיהוי קבצים רלוונטיים, ולאחריו שלב יצירת העריכה בפועל — כך שכל שלב ניתן לבדיקה בנפרד.
